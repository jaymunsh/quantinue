2ENUM·코드값영어 소문자 snake_case — trend·regime·sentiment·event_type·decision에 이어 side(buy/hold)·inv_type(aggressive)·bucket도 통일. 한글은 문서·UI 라벨로만. ※ 여기서 “ENUM”은 허용값 집합이라는 논리 표현 — Postgres 네이티브 ENUM 타입은 쓰지 않는다(값 추가마다 ALTER TYPE 마이그레이션 부담). 물리 타입은 전부 TEXT이고, 값 검증의 1차 책임은 백엔드(Pydantic Literal · core/schemas.py — event_type은 ontology.py가 정본), DB는 선택적으로 CHECK 제약만. 참조 테이블은 두지 않는다 — event_type도 예외 없이 TEXT (ENUM) 🔄 07-11 확정 (AI 제안 #12). 🆕 문서 규칙: ENUM 컬럼은 스키마 표 설명란에 허용값 전체를 필히 명시 — 값 목록이 미확정이면 ⚠️ ENUM 값 미확정 표시를 남기고 확정 즉시 업데이트문자열 비교 버그(trend 사건)의 재발 방지 — DB 값과 코드 상수가 항상 같은 어휘 · ENUM 타입의 마이그레이션 함정 회피

기각된 종목은 5사이클 쿨다운(재제안 억제) + 판정 캐시(기각뿐 아니라 통과도 캐시 — 아니면 같은 상황을 매 사이클 LLM에 다시 묻는다) + 미세 변동은 버킷으로 뭉갬(가격 1% · rsi 5p · trust 0.1). LLM을 실제로 부른 판정만 source=fresh.

8동시각 충돌 방지 (재실행 안전) + PK 규칙👤 정창욱✅ 해결 (07-11 확정)
추천안
“시각을 PK로 쓰지 말고, 사이클을 명시적 키로 쓰자”가 정공법. signal 테이블에 cycle_ts(그 사이클의 계획된 실행 시각 — 예: 30분 슬롯이면 정확히 09:00:00·09:30:00 …)를 두고 PK/유니크 = (ticker, cycle_ts) + ON CONFLICT (ticker, cycle_ts) DO NOTHING. created_at은 “행이 실제 만들어진 벽시계”로 그냥 기록만(밀리초 포함). ▸ 이러면 스케줄러가 한 사이클을 두 번 돌려도(재시작·중복 트리거) 같은 (ticker, cycle_ts)라 두 번째는 조용히 무시 = 멱등. 스케줄 섹션의 “멱등 append”가 이걸 가리킨다.
왜
지금은 created_at이 초 단위 문자열이라 같은 초에 재실행되면 PK가 충돌한다. 반대로 밀리초만 붙이면(초안) 충돌은 안 나지만 재실행 때마다 중복 행이 쌓인다 — “충돌 회피”와 “중복 방지”는 다른 문제고, 후자를 풀려면 “이 사이클의 정체”를 담는 키가 필요하다.
❓ 사이클
번호란
질문 답: 여기서 말한 “사이클 번호(cycle_id)”는 폐기된 옛 설계의 잔재다 — 원래 이은미 v3 이전엔 “1일차-3번째 판단” 식으로 정수 일련번호를 붙여 신호끼리 이었는데, 가리킬 마스터 테이블이 없어 반쪽 FK가 되는 문제로 이미 폐기(협의 현황판 ✅#1)하고 trade_date + src_*_at 방식으로 갈아탔다. 그래서 대안의 “사이클 번호를 PK에 포함”은 그 폐기된 정수 id의 부활이라 비추천이고, 위 추천안의 cycle_ts는 정수 번호가 아니라 “그 사이클의 계획 시각(타임스탬프)”이라 마스터 테이블 없이도 자기완결적이다.
대안
밀리초만 붙여 충돌 회피(중복 행은 감수) / 사이클 정수 번호를 PK에 포함(cycle_id 부활이라 비추천)
↳ ✅ 해결 (2026-07-11 확정) — 추천안에서 한 걸음 더 나가 3단 확정 + 컨벤션 규칙 8 신설: ① created_at은 전 테이블에서 PK 금지 — “DB에 언제 들어갔나” 확인용 기록으로 강등. ② signal 계열(사이클 append)은 대리키 PK + UNIQUE(ticker, cycle_ts) — 대리키 이름은 테이블 약어+_id(tb_news_signal→tns_id · tb_disclosure_signal→tds_id). id는 신분증, cycle_ts 유니크가 멱등 장치 — 대리키만으론 재실행 중복을 못 막으므로 반드시 세트. ③ 일봉 계열(tb_universe·tb_daily_pick·tb_technical)은 (날짜, ticker) 자연 복합 PK 유지 — “하루 1행” 규칙을 PK가 강제하는 방어 장치라, 대리키로 바꿔도 UNIQUE를 다시 걸어야 해 얻는 게 없음(ticker는 단독 PK만 금지 — 복합키 구성원·FK는 허용). 파급 반영 완료: 규칙 8 · 리네임 매핑표 · ERD · 테이블 총람 · 05/06 스키마 표(키 2필드 추가로 번호 +2) · 07의 src_*_at FK 대상이 created_at→cycle_ts로 교체. ※ tb_strategist_signals·tb_critic_verdict는 이미 id PK라 규칙 위반 없음 — 07 재실행 멱등이 필요해지면 같은 패턴(UNIQUE(ticker, cycle_ts))을 적용(이은미 확인 사항).

11Form 4 reason 템플릿👤 정창욱✅ 채택 (07-11)
❓ Form 4
란
질문 답: Form 4 = 미국 SEC(증권거래위) 공시 양식 중 “내부자 거래 보고서”다. 임원·이사·10% 이상 대주주가 자기 회사 주식을 사고팔면 거래 후 2영업일 내 반드시 제출한다. 우리가 이걸 보는 이유: “내부자가 자기 돈으로 자사주를 샀다”는 강력한 강세 신호(회사 사정을 가장 잘 아는 사람의 베팅)라서. 필드가 전부 숫자·코드(누가·매수/매도·주수·금액·거래코드 P=공개매수)여서 LLM 해석이 전혀 필요 없고 100% 코드로 스코어링한다 — LLM 지도에서 “Form 4 = LLM 0회”인 이유. 필요한가: 예 — 공시 신호(05)의 핵심 축 중 하나. 다만 8-K/10-Q(서술형, LLM 해석 필요)와 달리 가장 싸고 확실한 신호라 우선순위가 높다.
제안
Form 4의 reason은 빈 문자열 대신 코드가 템플릿 생성 (예: "Officer buy $1.2M, open market (code P)") — 값들이 이미 다 있으니 문장으로 조립만
왜
빈 문자열은 “근거 없음”과 “생성 실패”를 구분 못 하고, 우리 공통 계약인 “점수마다 근거 동봉”을 위반한다. 07·11이 나중에 “왜 이 종목이 샀지”를 되짚을 때 사람이 읽을 한 줄이 남는다.
대안
(상세)
reason=null 허용으로 계약을 완화 — ✗ 비추천: “점수마다 근거” 계약이 딱 한 곳(Form 4)에서만 깨지면 소비자(07 프롬프트·11 회고)가 “null이면 무시”라는 예외 분기를 따로 둬야 해서 오히려 복잡해진다. 템플릿 몇 줄이 예외 분기보다 싸다. 템플릿 문구 카탈로그(거래코드 P/S/A별 문장)는 정창욱이 05 구현 때 확정.
↳ ✅ 채택 (2026-07-11) — 코드가 reason 템플릿 생성 확정 (구현: 정창욱 — 05 구현 때 거래코드 P/S/A별 문장 카탈로그 확정). 무엇이 달라지나: 지금 설계대로면 Form 4 행의 reason이 빈 문자열이라 07 프롬프트·11 회고가 “근거가 없는 것”인지 “생성이 실패한 것”인지 구분 못 한다. 채택 후엔 이미 파싱해 둔 값(누가·매수/매도·금액·거래코드)을 문장으로 조립해 "Officer buy $1.2M, open market (code P)"(임원이 공개시장에서 자사주 120만 달러 매수) 같은 사람이 읽는 한 줄이 자동으로 남는다. LLM 호출은 여전히 0회(문자열 조립일 뿐)고, “점수마다 근거 동봉” 계약이 예외 없이 지켜져 소비자(07·11)의 null 예외 분기가 사라진다.
12event_type 마스터 테이블👤 정창욱 · 김지현 (구현: 정창욱)✅ 해결 (07-11 — 대안 채택)
쉽게
말하면
“사건 종류 이름표(8-K·form4·earnings…)를 코드에만 두지 말고, DB에 ‘공식 목록 테이블’로 하나 만들어 두자”는 것. 비유하면 — 지금은 사건 종류가 각 사람 수첩(코드·문서)에만 적혀 있어서, 누가 "form_4"라고 오타를 내도 DB가 순순히 받아준다. tb_event_type 테이블을 만들어 허용된 12종만 등록해두고 다른 테이블들이 이걸 FK로 참조하면, 목록에 없는 값은 DB가 저장 자체를 거부(=오타 방역). 덤으로 “form4 = 한글로 ‘내부자 거래’, 하드리스크 여부 = true” 같은 부가정보의 진실원천이 한 곳이 된다.
제안
tb_event_type 참조 테이블(code PK·label_ko·is_hard_risk·description·created_at) — 12종 seed는 05 섹션 표에 고정. + 앱 기동 시 코드(ontology.py의 Pydantic Literal)와 DB 테이블이 일치하는지 자동 검증, 어긋나면 기동 실패(둘이 몰래 갈라지는 것 방지). 본문 반영 완료.
👉 구현
메모
구현 주체가 사실상 나(PM)로 넘어온 상황 — 할 일은 3가지뿐이다: ① 마이그레이션에 tb_event_type 생성 + 12종 INSERT(seed) ② 4개 테이블(tb_news·tb_disclosure·각 signal)의 event_type 컬럼에 FK 걸기 ③ 앱 부팅 시 set(DB codes) == set(ontology Literal) 검사하는 짧은 정합 체크. 방향경향·발생원 같은 “예단” 컬럼은 일부러 뺐다(SESSION 기준) — label_ko·is_hard_risk·description만.
대안
CHECK 제약만 걸고 ontology.py를 진실원천으로 유지 — 테이블·seed 마이그레이션은 불필요하지만 한글 라벨 등 메타는 코드에만 남아 “DB만 봐선 뜻을 모른다”
↳ ✅ 해결 (2026-07-11 · 정창욱 확정) — 제안 기각, 대안 채택(방침 변경). tb_event_type 참조 테이블은 만들지 않는다 — event_type도 side·bucket 등 다른 ENUM과 똑같이 TEXT (ENUM)(규칙 2 · “예외 0개” 원칙). 진실원천은 ontology.py Pydantic Literal 하나 + 05 허용값 12종 표(문서 정본). DB 차단(FK 오타 방역)은 포기하는 대신 백엔드 검증이 1차 방어 — 마이그레이션·seed·기동 정합 검증이 전부 불필요해진다. + 신규 문서 규칙(규칙 2에 명문화): ENUM 컬럼은 스키마 표 설명란에 허용값 전체 필히 명시, 미확정이면 ⚠️ 표시 후 확정 시 업데이트 — 현재 ⚠️ 3곳: tb_order 유형·상태 / tb_fill 매수·매도 / tb_review_price_snapshots.source. 파급 반영 완료: 규칙 2·6 · ERD · 테이블 총람(폐기 표시) · 05 허용값 표 전환 · 4개 테이블 event_type 행 TEXT (ENUM)화 + 12종 명시.

14수집 주기 완화 (1차 MVP)👤 정창욱 · 김지현(스케줄러)✅ 채택 (07-11)
한 줄
요약
“뉴스를 5분마다 모으던 걸 1차에선 30분마다로 늦추자” — 그게 전부다. 왜 괜찮냐면, 최종 판단(07)이 2시간에 한 번만 도는데 뉴스를 5분마다 모아봐야 판단 한 번당 24개 스냅샷 중 1개만 읽히고 23개는 아무도 안 본다. 버려질 걸 만드느라 LLM 배치 호출·DB 행만 24배로 쌓는 셈. 수집 속도를 판단 속도에 맞춰 늦추는 것(경험칙: 수집 ≈ 판단 ÷ 2~4).
제안
“수집 주기 ≈ 판단 주기 ÷ 2~4” 채택. 1차(07=2시간): 뉴스 5분 → 30분, 공시 1시간 유지. 07이 1시간으로 좁히면 뉴스 15분, 10분 단계에서 뉴스 5분 복원(단계 동조 — 스케줄 로드맵 표). 유지할 것 2가지: ① 증분 수집(새 기사·새 accession만 LLM — 비용의 핵심 방어선) ② has_signal=false 빈 행(감사 균일성).
왜
기사별 LLM 분석은 증분이라 주기와 무관하지만, 배치 종합 LLM 호출·집계 연산·DB 성장은 주기에 정비례. 30분으로 늦추면 DB 성장·배치 호출 모두 1/6. 소규모 확인용 프로젝트에 5분은 과설계.
