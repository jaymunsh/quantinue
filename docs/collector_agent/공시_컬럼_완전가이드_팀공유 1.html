0.핵심 3줄 & 공시 vs 뉴스 결정적 차이

① 공시는 “1건이 곧 사실”입니다. SEC에 법적으로 제출된 공식 문서라, 뉴스처럼 여러 건을 신뢰가중으로 뭉치는 집계가 없습니다. 그래서 컬럼도 더 단출(20)해요.

③ 처리 방식이 3갈래. 8-K는 항목 라우팅+LLM, 10-Q/K는 XBRL 숫자+LLM 해석, Form 4는 100% 코드 스코어링.

2.공시 3축 — 8-K · 10-Q/K · Form 4

공시는 종류마다 성격이 달라서 세 가지 다른 방법으로 처리합니다. 핵심 원칙은 “숫자는 코드가, 해석은 LLM이”예요.

① 8-K — 속보성 사건 서류

회사에 갑자기 생긴 큰 일(대형 계약·임원 교체·인수합병 등)을 알리는 문서. 안에 항목 번호(Item)가 붙어 있어요.

  • 항목 라우터가 Item 번호로 사건 종류를 1차 판별합니다(예: Item 5.02 = 임원 변경).
  • 일부 Item은 기본 권한이 정해져 있어요 — 예: Item 3.01·4.01은 BLOCK_BUY(상폐·감사인 교체 등 위험).
  • 그 다음 LLM이 본문을 읽고 방향·강도·요약을 채웁니다.

② 10-Q / 10-K — 분기·연간 실적 서류

3개월(10-Q)·1년(10-K) 실적을 숫자로 정리한 “성적표”. 여기선 XBRL(재무 데이터 표준 형식)을 씁니다.

  • 코드가 XBRL에서 매출·이익 등을 뽑아 전년 대비(YoY)를 정확히 계산합니다.
  • LLM은 그 숫자가 좋은지 나쁜지 해석만 합니다 — 숫자 자체를 지어내게 두지 않아요.
  • 추가로 본문에서 going concern(존속 우려)·material weakness(중대한 결함) 같은 위험 문구를 코드가 별도 스캔합니다.

③ Form 4 — 내부자 거래 (100% 코드)

임원이 자기 회사 주식을 사고팔면 내는 문서. LLM 없이 코드만으로 점수를 매깁니다(규칙이 단순).

행위해석방향
공개시장 매수회사를 잘 아는 사람이 자기 돈으로 삼호재
공개시장 매도차익실현·현금화(사유 다양)약한 악재
베스팅·세금·옵션행사보상 제도에 따른 기계적 거래중립

직급이 높을수록(CEO·CFO) 가중치를 더 줍니다. 결과는 event_type=insider_trade로 공통 배관에 흘려보내요.

3.전달 계약 상세 — 전략가가 받는 JSON

# pack_disclosure_signal 이 만드는 전달 JSON
{
  "has_signal": true/false,     # ① 항상 존재

  # ② has_signal=true 일 때만 (판단 본문 + 근거 7개)
  "event_type": "ma",
  "importance_score": 0.8,
  "importance_score_reason": "...",   # 왜 그 강도인지 한 문장(LLM)
  "sentiment_score": 0.85,
  "sentiment_score_reason": "...",    # 왜 그 방향인지 한 문장(LLM)
  "risk_score": 0.1,
  "risk_score_reason": "...",        # 왜 그 위험인지 한 문장(LLM)

  "hard_block": false,       # ③ 항상 존재
  "summary": "..."         # has_signal=true 일 때만
}

뉴스와 규칙이 같습니다: has_signal·hard_block은 항상, 나머지는 조건부. 차단 시 summary 앞에 [차단: 사유]가 붙습니다. 개명은 importance→importance_score·confidence→confidence_score이고, 각 _score 옆에 근거 한 문장({score}_reason)이 함께 실려요(공시엔 source_trust가 없으니 trust_score 개명도 없음).

뉴스에 있는데 공시엔 없는 3필드: article_count(공시는 1건)·peak_importance_score(집계 안 함)·trust_score(출처가 SEC라 이미 신뢰). 대신 _score마다 근거 컬럼 3개가 붙어 10필드예요.

4.전달 JSON 실물 예시 4개 (+해설)

아래는 실제 pack_disclosure_signal을 돌려 뽑은 결과입니다. 3축을 각각 보여줄게요.

예시 1 · 8-K 인수 발표 (호재)

{
  "has_signal": true,
  "event_type": "ma",
  "importance_score": 0.8,
  "importance_score_reason": "Large all-cash acquisition materially reshapes the business.",
  "sentiment_score": 0.85,
  "sentiment_score_reason": "Accretive deal at a strategic premium; clearly bullish.",
  "risk_score": 0.1,
  "risk_score_reason": "Cash-funded, no financing or dilution risk.",
  "hard_block": false,
  "summary": "Microsoft agreed to acquire a cybersecurity firm for $2.1B in cash."
}

강도 0.8·방향 0.85(강한 호재)·위험 낮음. 공시는 확신도(certainty)가 대개 High(0.9)라 호재 감쇠가 거의 없어 방향이 또렷합니다.

예시 2 · 10-Q 실적 호조

{
  "has_signal": true,
  "event_type": "earnings",
  "importance_score": 0.7,
  "importance_score_reason": "Quarterly print with a clear guidance beat moves the stock.",
  "sentiment_score": 0.72,
  "sentiment_score_reason": "11% YoY revenue growth above guidance is positive.",
  "risk_score": 0.2,
  "risk_score_reason": "Solid results, but usual macro downside remains.",
  "hard_block": false,
  "summary": "Revenue up 11% YoY, beating guidance; services at record high."
}

YoY(전년대비) 숫자는 코드가 계산하고, “가이던스 상회 = 호재”라는 해석은 LLM. risk 0.2는 실적이 좋아도 남는 통상적 하방.

예시 3 · Form 4 내부자 매수 (코드 스코어링)

{
  "has_signal": true,
  "event_type": "insider_trade",
  "importance_score": 0.5,
  "importance_score_reason": "",
  "sentiment_score": 0.65,
  "sentiment_score_reason": "",
  "risk_score": 0.05,
  "risk_score_reason": "",
  "hard_block": false,
  "summary": "CEO purchased 50,000 shares on the open market."
}

LLM 없이 코드가 만든 신호예요. CEO의 공개시장 매수라 완만한 호재(0.65). 확신도가 Medium(0.6)이라 방향이 예시1보다 살짝 눌립니다. {score}_reason은 빈 문자열 — Form 4 코드 스코어링은 근거 문장을 LLM이 생성하지 않아요.

예시 4 · hard_block (상장폐지 통지)

{
  "has_signal": true,
  "event_type": "delisting_halt",
  "importance_score": 0.9,
  "importance_score_reason": "Delisting determination is an existential, top-severity event.",
  "sentiment_score": 0.05,
  "sentiment_score_reason": "Losing the exchange listing is strongly bearish.",
  "risk_score": 0.95,
  "risk_score_reason": "Imminent delisting threatens liquidity and going concern.",
  "hard_block": true,
  "summary": "[차단: BLOCK_BUY: delisting notice (Item 3.01)] Received delisting determination from the exchange."
}

점수와 무관하게 hard_block:true → 전략가는 즉시 매수 제외. 사유가 요약 맨 앞에 자동 삽입됩니다. 악재(0.05)는 감쇠 없이 또렷.

번외 · 신호 없음

{ "has_signal": false, "hard_block": false }

7.★ event_type 12종 완전 카탈로그

event_type은 공시와 뉴스가 공유하는 단일 목록입니다(ontology.py가 유일한 진실원천). 과세분화를 막으려고 11종 + other = 12종으로 통일했어요(너무 잘게 나누면 표본이 부족해 백테스트가 무의미해짐). LLM은 이 목록으로만 답하도록 Pydantic Literal로 강제됩니다.

8.공식 상세 — 공시판 (한글 + 숫자)

공시는 기사 여러 건을 뭉치는 집계가 없어서 뉴스보다 공식이 단순합니다. 뉴스의 신뢰가중 평균·peak·source_trust·grade_score가 전부 없어요. 딱 세 개면 됩니다.

① _n10 — 0~10을 0~1로 (뉴스와 동일)

결과 = (값을 0~10 사이로 자름) ÷ 10, 소수점 둘째 반올림
# 7 → 0.7,  9 → 0.9,  13 → 1.0(잘림)

② 확신도 — certainty_level을 숫자로

confidence_score = {High: 0.9, Medium: 0.6, Low: 0.3} 에서 조회
# 뉴스는 이걸 '기사 무게'의 한 요소로 썼지만,
# 공시는 1건이라 이 값이 곧 confidence_score 컬럼이 됩니다.
예시 · certainty=High → confidence_score 0.9 · Medium → 0.6

③ _senti_score — 방향을 0~1로 (호재만 감쇠, 뉴스와 동일)

방향강도 = 부호(sentiment) × importance_score      # 예: 호재(+1) × 0.8 = +0.8
만약 방향강도 > 0:  방향강도 = 방향강도 × confidence_score   # 호재만 감쇠
sentiment_score = 0.5 + 0.5 × 방향강도
호재 예시 · 방향강도 +0.8, confidence_score 0.9 → 0.8×0.9=0.72 → 0.5+0.36 = 0.86 (≈예시1의 0.85)
악재 예시 · 방향강도 -0.9 (감쇠 없음) → 0.5+0.5×(-0.9) = 0.05 (예시4)

공시는 확신도가 대개 High(0.9)라 호재 감쇠 폭이 작습니다. 그래서 뉴스보다 방향이 또렷하게 나오는 편이에요. 나머지 파생(importance_score=_n10(...)·risk_score=_n10(...))은 그대로입니다.

정리. 공시 컬럼은 “집계”가 없어 1건 신호를 0~1로 정규화 + 호재 감쇠가 전부입니다. 뉴스편 §7의 ④신뢰가중평균·⑤peak·⑥방향집계가 공시엔 아예 없다는 게 핵심 차이예요.

9.워크드 예제 — 8-K 한 건이 결과가 되기까지

MSFT가 “인수 계약 체결(8-K)”을 냈다고 합시다. LLM이 이렇게 판단했다고 하죠.

LLM 산출(원시)값
event_typema
sentiment(방향)positive
importance_score(0~10)8
risk_score(0~10)1
certainty_levelHigh

계산

confidence_score = High → 0.9
importance_score = _n10(8) = 0.8
risk_score  = _n10(1) = 0.1
방향강도    = +1 × 0.8 = +0.8 → ×0.9(호재 감쇠) = 0.72
sentiment_score = 0.5 + 0.5×0.72 = 0.86
hard_block  = 최종권한이 BLOCK류? → 아님 → 0

여기에 원문 식별(filing_title·filing_no·filed_at)과 _score별 근거(importance_score_reason·sentiment_score_reason·risk_score_reason)·요약·키워드가 붙어 20컬럼이 되고, 그중 10개만 뽑아 §4 예시1 JSON으로 나갑니다. (예시1 저장값은 0.85로 반올림 표기 — 실제 계산은 0.86 근방, round(2) 기준.)

10.FAQ & 코드 위치

Q. 공시는 왜 trust_score·grade_score가 없나요?

출처가 SEC(정부 공식 제출)라 매체 신뢰를 따질 필요가 없습니다. 이미 “법적 사실”이거든요. 그래서 신뢰는 confidence_score(LLM 확신도) 하나로 충분합니다.

Q. 공시는 왜 peak_importance_score·article_count가 없나요?

둘 다 여러 건을 뭉칠 때 필요한 개념입니다. 공시는 1건이 곧 하나의 사실이라 집계·건수 개념이 없어요.

Q. Form 4는 LLM을 안 쓰는데 event_type이 어떻게 붙나요?

코드가 규칙으로 insider_trade를 직접 지정합니다(매수/매도/베스팅 판별 + 직급 가중). 그 뒤 공통 배관을 그대로 타요.

Q. disclosure_ref는 공시 컬럼인가요?

아니요, 그건 뉴스 컬럼입니다. 뉴스가 같은 종목·같은 날 공시와 매칭되면 그 공시의 filing_no를 참조해 사실성을 보강하는 용도예요(뉴스편 참고).
