② 점수는 다 0~1. 방향(호재/악재)·강도·위험·신뢰가 전부 같은 자로 재져 있어 비교가 쉽습니다.
③ 시끄러운 루머는 자동으로 힘을 잃습니다. 모든 집계가 “믿을수록 크게”라는 신뢰가중이라, 저신뢰 기사는 가중치가 0에 수렴해요.

1.두 개의 층 — 저장(29) vs 전달(15) 아키텍처

“컬럼이 너무 많다”는 인상은 대부분 두 층을 하나로 착각해서 생깁니다. 우리 뉴스 데이터는 목적이 다른 두 개의 층으로 나뉘어 있어요.

왜 굳이 둘로 나눴나?

  • 전략가의 인지 부담을 줄이려고. 판단에 필요한 최소한만 넘기면, LLM이든 사람이든 핵심에 집중합니다. 29개를 다 던지면 노이즈가 됩니다.
  • 감사·재현·백테스트를 위해. 나중에 “이 신호 왜 이렇게 나왔지?”를 되짚으려면 원문·근거·중간 점수가 남아 있어야 합니다. 그건 전략가가 볼 필요는 없지만 우리에겐 필요해요.
  • 계약과 구현을 분리하려고. 저장 스키마(내부 구현)는 필요하면 늘리거나 줄일 수 있지만, 전달 계약(외부 인터페이스)은 안정적으로 유지됩니다. 둘을 묶으면 내부를 바꿀 때마다 전략가 코드가 깨져요.
비유. 저장 29컬럼은 재료 창고, 전달 15필드는 손님상에 나가는 완성 요리입니다. 창고가 크다고 손님이 재료를 다 보는 게 아니죠. 창고는 “이 요리 왜 이렇게 만들었지?”를 나중에 검증하려고 있는 겁니다.

파이프라인 위치: 수집(RSS) → 사전필터 → LLM 분석 → build_news_bundle(집계) → tb_news 저장(29) → pack_news_signal(전달 15). 저장과 전달은 이 흐름의 마지막 두 단계입니다.

1) 수집 — Google News RSS

종목명으로 구글 뉴스 RSS를 긁어 최신 기사를 모읍니다. API 키가 필요 없고, 각 기사엔 RSS <source> 태그가 있어 어느 언론사인지 바로 알 수 있어요. 이 단계에서 원문 식별 컬럼(news_title·source·published_at·ref)의 원천이 확보됩니다.

2) 출처 등급 매기기 — 도메인 기반 3단계

기사를 쓴 언론사 도메인을 정책 파일(news_trust_policy.yaml)과 대조해 등급을 매깁니다. 이게 나중에 grade_score가 돼요.

등급뜻도메인 예grade_score
ALLOW검증된 주류 언론 — 통과reuters.com · bloomberg.com1.0
GRAY애매/미등록/소셜 — 받되 조심개인 블로그 · x.com · reddit0.6
BLOCK신뢰 불가 — 차단알려진 가짜뉴스 도메인0.0

주의: 이 ALLOW/GRAY/BLOCK은 출처 신뢰 등급일 뿐, 매매 권한의 WATCH_ONLY 같은 것과는 완전히 별개입니다.

3) 사전필터 — LLM 부르기 전에 쓰레기 치우기

LLM 호출은 시간·비용이 듭니다. 그래서 그 전에 ①BLOCK 도메인 기사와 ②명백한 노이즈(광고성 키워드 등)를 미리 버립니다. 키워드 화이트/블랙리스트로 걸러요. — “비싼 AI를 부르기 전에 값싼 규칙으로 먼저 거른다”는 원칙.

4) LLM 분석 — 기사 한 건을 구조화 판단

살아남은 기사를 Pydantic AI로 분석합니다. LLM이 자유 서술이 아니라 정해진 스키마(필드가 고정된 형태)로 답하도록 강제해요. 여기서 기사별로 event_type·sentiment·importance·risk·is_confirmed·source_trust·summary·keywords·reason이 나옵니다.

핵심 원칙 — 숫자는 코드, 해석은 LLM. LLM에게 “매출이 얼마다” 같은 사실 숫자를 지어내게 두지 않습니다. LLM은 “이게 좋은 소식인가/얼마나 확실한가” 같은 해석·판단만 담당해요.

여기까지가 기사 1건 단위입니다. 이 개별 판단들을 여러 건 → 한 종목의 신호 1개로 합치는 게 다음 단계(build_news_bundle, §7 공식)이고, 그 결과가 29컬럼입니다.

2.전달 계약 상세 — 전략가가 실제 받는 JSON

전략가에게 넘어가는 건 pack_news_signal(dict) → dict가 만드는 납작한 JSON입니다. 구조는 이렇게 3덩어리예요.

# pack_news_signal 이 만드는 전달 JSON의 뼈대
{
  "has_signal": true/false,        # ① 항상 존재 — 신호가 있나?

  # ② has_signal=true 일 때만 붙는 12개 (판단 본문)
  "article_count":   3,       # 뭉쳐진 기사 수 (저장의 news_count)
  "event_type":      "product_deal",
  "importance_score": 0.63,    # 저장의 importance 를 개명
  "peak_importance_score":  0.70,
  "sentiment_score":  0.64,
  "risk_score":      0.06,
  "trust_score":     0.84,    # 저장의 source_trust 를 개명

  # _score마다 왜 그 값인지 한 문장 근거 (배치 종합 LLM 생성)
  "importance_score_reason":      "...",
  "peak_importance_score_reason": "...",
  "sentiment_score_reason":       "...",
  "risk_score_reason":            "...",
  "trust_score_reason":           "...",

  "hard_block": false,          # ③ 항상 존재 — 즉시 매수 차단?
  "summary": "..."            # has_signal=true 일 때만
}

설계 포인트 4가지 (주니어가 놓치기 쉬운 것)

포인트내용
항상 있는 2필드has_signal·hard_block은 신호가 없어도 항상 옵니다. 전략가가 “필드 존재 여부”로 분기하지 않고 불리언 값으로 안전하게 분기하도록.
조건부 13필드점수 7개 + _score 근거 5개 + summary는 has_signal=true일 때만 존재. 신호가 없으면 0.00 같은 가짜 값 대신 아예 생략 — “0점”과 “측정 안 함”을 혼동하지 않게.
_score 근거 5개각 점수마다 {score}_reason이 붙습니다(importance_score_reason·peak_importance_score_reason·sentiment_score_reason·risk_score_reason·trust_score_reason). “왜 그 값인지”를 배치 종합 LLM이 한 문장으로 생성 — 전략가가 점수의 근거를 바로 확인.
개명전달 계약에서만 importance→importance_score, peak_importance→peak_importance_score, source_trust→trust_score로 이름을 바꿉니다. 저장 컬럼·LLM 스키마는 그대로예요(파급 최소화). 이름 뒤 _score로 “이건 0~1 점수”임을 명확히.
hard_block 시 summary차단이 걸리면 summary 맨 앞에 [차단: 사유]가 자동으로 붙습니다(_summary_with_block). 전략가가 “왜 막혔나”를 요약 첫 줄에서 바로 보게.
공시(pack_disclosure_signal)와 비교. 공시 전달은 7필드(has_signal·event_type·importance_score·sentiment_score·risk_score·hard_block·summary)로 더 단출합니다. 뉴스에만 있는 3개(article_count·peak_importance_score·trust_score)는 “여러 기사를 뭉쳤다”는 뉴스 고유 성격 때문이에요. 공시는 1건이 곧 사실이라 건수·매체신뢰·희석 개념이 필요 없습니다.

3.전달 JSON 실물 예시 3개 (+ 필드별 해설)

아래 JSON은 설명용으로 지어낸 게 아니라 실제 pack_news_signal을 돌려 뽑은 결과입니다. 세 가지 대표 상황을 보여줄게요.

예시 1 · 확정된 호재 (기사들이 서로를 뒷받침)

엔비디아(NVDA)에 로이터·블룸버그의 확정 기사 2건 + 저신뢰 블로그 1건이 들어온 상황(§8 워크드 예제의 최종 산출물).

{
  "has_signal": true,
  "article_count": 3,
  "event_type": "product_deal",
  "importance_score": 0.63,
  "peak_importance_score": 0.7,
  "sentiment_score": 0.64,
  "risk_score": 0.06,
  "trust_score": 0.84,
  "importance_score_reason": "Two high-trust outlets report a large multi-year deal, lifting weighted intensity.",
  "peak_importance_score_reason": "The strongest verified article (Reuters) alone reaches 0.7, above the average.",
  "sentiment_score_reason": "Clearly positive but dampened by moderate certainty, so not a strong bullish reading.",
  "risk_score_reason": "No litigation, regulatory, or downside triggers appear in any article.",
  "trust_score_reason": "Reuters and Bloomberg dominate the weighting; the low-trust blog barely contributes.",
  "hard_block": false,
  "summary": "NVIDIA signed a multi-year GPU supply agreement with a major cloud provider; analysts raised price targets."

필드이 값이 전략가에게 말하는 것
event_type: product_deal가장 신뢰가중이 큰 대표 기사가 “대형 계약/제품” 유형이라는 뜻.
sentiment_score: 0.640.5(중립)보다 위 → 호재. 다만 확신도(0.45)로 한 번 눌려서 “강한 호재(0.9)”까지는 아닌 완만한 호재.
importance_score: 0.63 · peak 0.7평균 강도 0.63인데 peak가 더 높음 → 믿을 만한 기사 중 더 센 게 하나 있다는 신호.
trust_score: 0.84매체 신뢰가 높음(로이터·블룸버그 주도). 블로그의 영향은 거의 없음.
risk_score: 0.06 · hard_block:false하방 위험 거의 없음, 즉시 차단 사유도 없음.

예시 2 · 저신뢰 루머 위주 (약한 신호)

확인 안 된 블로그 인수설이 4건. 신호는 있지만 믿을 근거가 얇은 상황.

{
  "has_signal": true,
  "article_count": 4,
  "event_type": "ma",
  "importance_score": 0.35,
  "peak_importance_score": 0.35,
  "sentiment_score": 0.55,
  "risk_score": 0.2,
  "trust_score": 0.35,
  "importance_score_reason": "Only unconfirmed blog chatter, so weighted intensity stays modest.",
  "peak_importance_score_reason": "No verified article qualifies, so peak collapses to the average.",
  "sentiment_score_reason": "Mildly positive framing but heavily discounted by low certainty.",
  "risk_score_reason": "Speculative M&A talk carries some downside if it is denied.",

  "hard_block": false,

여기서 배울 점. 기사가 4건이나 되고 “인수설”이라 자극적이지만 — trust_score 0.35, sentiment_score 0.55(간신히 호재쪽), 그리고 peak_importance_score(0.35) = importance_score(0.35)입니다. peak가 평균과 같다는 건 “신뢰검증된 강한 기사가 하나도 없다”는 뜻이에요(§7-⑥ 참고). 전략가는 이 조합을 보고 “화제성은 있으나 근거가 약함 → 신중”으로 읽습니다.

예시 3 · hard_block 발동 (거래정지·상장폐지 우려)

거래소에서 상장폐지 통지를 받은 상황. 점수와 무관하게 즉시 매수 차단.

{
  "has_signal": true,
  "article_count": 2,
  "event_type": "delisting_halt",
  "importance_score": 0.8,
  "peak_importance_score": 0.85,
  "sentiment_score": 0.05,
  "risk_score": 0.9,
  "trust_score": 0.7,
  "importance_score_reason": "A delisting notice is a high-impact corporate event, driving intensity up.",
  "peak_importance_score_reason": "The exchange-notice article is verified and stands as the strongest single signal.",
  "sentiment_score_reason": "Strongly negative; bad news is not softened by certainty.",
  "risk_score_reason": "Delisting threatens near-total downside, so risk is near maximum.",
  "trust_score_reason": "Reported by mainstream outlets, giving solid media trust.",

  "summary": "[차단: delisting risk (Item 3.01)] Company received a delisting notice from the exchange."

필드해설
hard_block: true가장 중요한 신호. 점수가 어떻든 매수 금지. 전략가는 이 하나로 매수 후보에서 제외합니다.
summary 앞의 [차단: …]차단 사유가 요약 맨 앞에 자동 삽입 → “왜 막혔는지”를 즉시 파악.

risk_score: 0.9하방 위험 매우 높음.

번외 · 신호 없음 (오늘 이 종목엔 뉴스 신호가 없음)

{
  "has_signal": false,
  "hard_block": false
}

6.헷갈리기 쉬운 3쌍 정면돌파

“이해가 안 된다”의 진짜 원인은 비슷해 보이는 컬럼들입니다. 세 쌍만 구분하면 됩니다.

쌍 1 · importance_score ↔ sentiment_score — 강도 vs 방향
importance_score는 “얼마나 큰 일이냐”(부호 없음). sentiment_score는 “좋은 일이냐 나쁜 일이냐”(방향). 큰 악재는 강도 높고 방향 점수는 낮아요. 독립된 축이라 하나로 못 합칩니다.
쌍 2 · confidence_score ↔ trust_score ↔ grade_score — 세 종류의 “신뢰”
· confidence_score = 우리 집계의 확신(근거가 두꺼운가) — 기사별 신뢰치의 평균.
· trust_score = LLM이 내용을 읽고 본 매체 신뢰(사후).
· grade_score = 코드가 도메인만 보고 매긴 등급(사전, 규칙: ALLOW/GRAY/BLOCK).
일부러 따로 둡니다 — LLM 판단과 규칙 판단을 이중 교차검증(감사축)하려고요. 하나가 속아도 다른 하나가 잡습니다.
쌍 3 · importance_score ↔ peak_importance_score — 평균 vs 최고
importance_score는 전체 기사의 평균 강도, peak_importance_score는 가장 센 신뢰기사 1건. 평균이 밋밋해도 단독 대형 신호가 있으면 peak가 튀어요. peak ≥ importance_score는 항상 성립(peak가 importance를 포함해 max를 취하므로).

7.공식 상세 — 영어 코드를 한글로 풀고 숫자로 확인

코드에 있는 함수들을 영어 그대로 두지 않고, ①한글 정의 → ②단계별 숫자 예시 → ③왜 이렇게 순으로 풀겠습니다. 여섯 개면 전부 계산됩니다.

① _n10 — “0~10 점수를 0~1로 줄이기”

LLM은 보통 0~10으로 답합니다. 우리는 모든 점수를 0~1로 통일해요. 함수 이름 _n10은 “normalize from 10”의 뜻입니다.

# 한글로 풀면:
결과 = (값을 0~10 사이로 자른 뒤) ÷ 10, 소수점 둘째 자리 반올림
# 원래 코드: round(clamp(v, 0, 10) / 10, 2)
예시 · 7 → 0.7 · 9 → 0.9 · 13 → 1.0(10으로 잘림) · -2 → 0.0(0으로 잘림)

왜: 방향·강도·위험·신뢰가 전부 같은 0~1 자로 재져야 서로 곱하고 비교하기 쉽습니다.

② _news_conf — “기사 한 건의 신뢰 무게”

여러 기사를 합칠 때 쓰는 기사별 가중치(무게)입니다. 세 요소의 곱이에요. 하나라도 낮으면 무게가 확 줄어듭니다.

# 한글로 풀면:
기사 무게 = 확신도 × 확인계수 × 매체신뢰

  확신도   (LLM certainty) : High=0.9 · Medium=0.6 · Low=0.3
  확인계수 (is_confirmed): 확인된 사실이면 1.0, 아직 소문이면 0.5
  매체신뢰 (source_trust): LLM이 본 0~1 값
예시 · 확실(High) + 확인됨 + 매체0.9 → 0.9 × 1.0 × 0.9 = 0.81
· 애매(Low) + 소문 + 매체0.3 → 0.3 × 0.5 × 0.3 = 0.045 ← 무게가 거의 0
주의(코드 함정). 매체신뢰가 0.0(완전 무신뢰)일 때 파이썬 x or 1.0 관용구를 쓰면 0.0을 1.0으로 오인합니다. 그래서 코드는 st if st is not None else 1.0으로 None만 1.0 취급하고 0.0은 그대로 곱합니다.

③ _senti_score — “방향을 0~1로 (비대칭 감쇠)”

“방향×강도”(-1~+1)를 0~1 점수로 바꿉니다. 핵심은 호재만 확신도로 깎고, 악재는 안 깎는 비대칭성이에요.

# 한글로 풀면:
방향강도 = (-1 ~ +1 사이로 자름)         # 양수=호재, 음수=악재
만약 방향강도 > 0 (호재):  방향강도 = 방향강도 × 확신도   # 호재만 감쇠
sentiment_score = 0.5 + 0.5 × 방향강도    # 악재(-)는 그대로
호재 예시 · 방향강도 +0.635, 확신도 0.45 → 0.635×0.45 = 0.286 → 0.5 + 0.5×0.286 = 0.64
악재 예시 · 방향강도 -0.8 (감쇠 없음) → 0.5 + 0.5×(-0.8) = 0.1

왜: 근거 약한 장밋빛 소문에 속아 매수하는 게 가장 위험합니다. 그래서 호재는 “정말?” 하고 한 번 더 의심(중립 0.5쪽으로 감쇠)하고, 악재(경고)는 흐리지 않습니다. — “가장 보수적 채택” 사상.

④ 신뢰가중 평균 — “믿을수록 크게 반영한 평균”

importance_score·risk_score·trust_score·grade_score가 전부 이 패턴으로 계산됩니다(예: trust_score = Σ(source_trust × 무게)/tw — 여기서 source_trust는 기사별 입력).

# 한글로 풀면:
지표 = Σ(각 기사 값 × 그 기사 무게) ÷ Σ(무게)
# 무게 = ②의 _news_conf
# 예외: 모든 기사 무게가 0이면 → 등가중(전부 1)으로 폴백해 0 붕괴 방지
예시(importance_score) · 강도[0.7, 0.5, 0.9], 무게[0.81, 0.48, 0.045], 무게합 1.335
→ (0.7×0.81 + 0.5×0.48 + 0.9×0.045) ÷ 1.335 = 0.8475 ÷ 1.335 = 0.63

주목: 세 번째 기사는 강도가 0.9로 제일 높지만 무게가 0.045라 결과에 거의 기여하지 못합니다.

⑤ peak_importance_score — “신뢰검증된 기사 중 가장 센 것”

# 한글로 풀면:
신뢰기사 = (확인됨 이거나 매체신뢰≥0.6 이거나 등급 ALLOW) 인 기사
peak = max( importance_score, 신뢰기사들의 강도… )
예시 · importance_score 0.63, 신뢰기사 강도 [0.7, 0.5] (저신뢰 0.9는 제외) → max(0.63, 0.7, 0.5) = 0.70

왜: 평균에 묻힌 단독 대형 신호를 살리되, 저신뢰 매체의 과장(0.9)이 가짜 peak를 세우지 못하게 신뢰기사만 후보로 씁니다. importance_score를 항상 후보에 넣어 peak ≥ importance_score를 보장합니다.

⑥ 방향 집계 & 라벨 규칙 — “강한 악재는 우선한다”

기사별 방향강도를 합칠 때, 아주 나쁜 기사 1건이 있으면 평균 대신 그쪽을 택합니다(보수).

# 방향 집계:
만약 가장 나쁜 기사 ≤ -0.7 :  집계값 = 그 가장 나쁜 값     # 강한 악재 우선
아니면                      집계값 = 신뢰가중 평균
sentiment_score = ③(_senti_score)(집계값, 확신도)

# 라벨(sentiment) 규칙:
강한 악재 있음                → negative
호재무게·악재무게 각 30%↑     → mixed          # 둘 다 유의미
점수 > 0.5 → positive / < 0.5 → negative / = 0.5 → neutral
예시 · 방향강도 [+0.7,+0.5,+0.9](강한 악재 없음) → 가중평균 0.635 → _senti_score(0.635, 0.45) = 0.64 → positive

라벨을 최종 점수에서 파생하기 때문에 “라벨은 positive인데 점수는 0.4” 같은 모순이 생기지 않습니다.

정리. 위 6개 공식이면 tb_news의 점수 컬럼 숫자가 전부 재현됩니다(근거 _reason 5개는 LLM 생성). grade_score도 ④와 같은 신뢰가중 평균이고, 값만 {ALLOW:1, GRAY:.6, BLOCK:0}을 씁니다.

위 29컬럼 중 15개만 뽑고 importance→importance_score, peak_importance→peak_importance_score, source_trust→trust_score로 개명하면(+ _score마다 근거 5개 첨부) → §3 예시1의 JSON이 그대로 나옵니다.

이 예제의 교훈. A3는 강도가 9로 제일 높지만 — 소문·저신뢰·GRAY라 무게가 0.045로 쪼그라들어 ❶ 평균을 거의 못 움직이고 ❷ peak 후보에서 제외되며 ❸ 대표 기사도 못 됩니다. 신뢰 컬럼이 3개인 이유가 이거예요 — 시끄러운 과장이 자동으로 걸러지는 구조.

9.자주 묻는 질문(FAQ) & 흔한 오해

Q. 컬럼이 29개나 되는데 좀 줄이면 안 되나요?

전략가가 읽는 건 15개뿐이라 “많다”의 체감 주체가 사실 없습니다. 저장 29컬럼은 감사·백테스트용 창고라, 줄이면 “왜 이 신호가 나왔나”를 되짚는 능력을 잃습니다. “비슷해 보이는” 컬럼도 실은 §6의 서로 다른 축이고, _score_reason 5개는 각 점수의 근거라 중복이 아닙니다.

Q. confidence_score랑 trust_score랑 grade_score, 결국 다 “신뢰” 아닌가요?

아니요. 우리 확신 / LLM이 본 매체 / 코드가 매긴 도메인등급으로 출처가 다릅니다(§6 쌍2). 특히 뒤 둘을 나눈 건 LLM과 규칙이 서로를 교차검증하게 하려는 감사 설계예요.

Q. ALLOW가 붙으면 “사도 된다”는 뜻인가요?

아닙니다. ALLOW/GRAY/BLOCK은 출처(언론사) 신뢰 등급일 뿐, 매수 허가가 아닙니다. 매수 판단은 전략가 몫이고, 우리 쪽 “차단”은 hard_block이 담당합니다.

Q. sentiment_score가 0.5면 “호재 절반”인가요?

0.5는 중립입니다. 방향이 없다는 뜻이지 “50% 호재”가 아니에요. 강도(중요도)는 importance_score가 따로 표현합니다.

Q. 왜 저신뢰 기사도 버리지 않고 집계에 넣나요?

완전히 버리는 대신 무게를 아주 작게 줍니다(§7-②). 그래야 confidence_score가 “근거가 얇다”를 정직하게 낮게 반영할 수 있어요. 아예 버리면 그 사실이 사라집니다. (단, 사전필터에서 명백한 노이즈·BLOCK 도메인은 LLM 전에 제거됩니다.)

Q. peak_importance_score가 importance_score랑 똑같이 나오면?

신뢰검증된 강한 기사가 없다는 뜻입니다(§3 예시2). peak가 importance_score로 수렴했다는 건 “평균 이상으로 튀는 믿을 만한 단독 신호가 없음”이에요.
