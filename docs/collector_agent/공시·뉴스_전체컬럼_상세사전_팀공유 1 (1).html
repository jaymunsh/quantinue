수집은 “증분”으로 — 새 것만 처리

2,000종목을 매 사이클 다 다운로드·분석하면 비용이 폭발합니다. 그래서 메타레벨 증분으로 돌아요.

  • processed_filings(accession PK)로 처리 여부 판정 → 새 accession일 때만 문서 다운로드·LLM·저장.
  • 뉴스도 같은 순회에서 기사별 유니크 링크로 중복 제거(재분석 방지).

3.공시 3축 — 8-K · 10-Q/K · Form 4

공시는 종류마다 성격이 달라 세 방법으로 처리합니다.

① 8-K — 속보성 사건

큰 계약·임원 교체·인수합병 등. 안의 항목 번호(Item)로 사건 종류를 1차 판별(예: 5.02=임원변경). 일부 Item은 기본 권한이 정해짐 — 3.01·4.01은 BLOCK_BUY(상폐·감사인 교체). 그 뒤 LLM이 본문을 읽고 방향·강도·요약을 채웁니다.

② 10-Q / 10-K — 분기·연간 실적

XBRL(재무 표준)에서 코드가 매출·이익을 뽑아 전년대비(YoY)를 계산, LLM은 해석만. 추가로 본문에서 going concern·material weakness 같은 위험 문구를 코드가 별도 스캔합니다.

③ Form 4 — 내부자 거래 (100% 코드)

임원이 자기 회사 주식을 사고팔면 제출. LLM 없이 코드 규칙: 공개시장 매수=호재 / 매도=약한 악재 / 베스팅·세금·옵션행사=중립, 직급 가중. 결과는 event_type=insider_trade로 공통 배관을 탑니다.

사전필터 & LLM

LLM 호출 전에 ①BLOCK 도메인 ②명백한 노이즈(광고성 키워드)를 미리 버립니다(비용 절감). 살아남은 기사를 Pydantic AI가 고정 스키마로 분석합니다.

키워드 필터동작예
화이트리스트 (우선)확실한 이벤트 키워드 → 통과 보장acquisition · earnings · FDA approval
블랙리스트광고·홍보성 노이즈 → drop“guaranteed returns” · “must-buy penny stock”
둘 다 아님통과(pass) → LLM 분석으로일반 기사

화이트리스트가 블랙리스트보다 우선합니다(false negative 방지 — 진짜 이벤트가 광고성 단어 하나로 버려지지 않게).

LLM이 기사 1건에서 뽑는 것 (고정 스키마)

Pydantic Literal로 정해진 값만 답하도록 강제합니다. 자유 서술이 아니라 구조화 출력이라 파이프라인이 안정적이에요.

필드뜻
reasoning[CoT] 헤드라인 핵심을 단계적으로 추론(추론 강제)
certainty_level단정 보도(High) vs 모호·추측(Low) → 무게의 확신도 요소
event_type · sentiment · importance · risk_score사건 유형·방향·강도(0~10)·위험(0~10)
is_confirmed확정 사실인가 (‘~알려져/전망/소문’이면 False) → 무게의 확인계수
source_trust출처 신뢰 0~1 (정책 등급 + 본문 톤 반영). 기사별 입력값으로 집계 trust_score의 재료.
summary · keywords요약·키워드(전부 영어). 점수별 근거(_score_reason)는 배치 종합 LLM이 생성.

순서본다판단

7.대표 기사 vs 집계 — 값의 출처

뉴스 컬럼은 세 갈래 출처로 나뉩니다. 어느 값이 어디서 오는지 알면 헷갈리지 않아요.

출처 방식컬럼설명
대표 기사 1건event_type + 원문식별(news_title·source·published_at·ref)대표 = importance_score×무게(신뢰가중) 최고 기사. 분류는 하나의 일관된 기사에서.
LLM 배치 종합summary · keywords · *_score_reason기사 2건 이상이면 전 기사를 1회 LLM 호출로 종합(“인수·소송 포함, 종합 호재” 식)하고 각 점수 근거를 한 문장씩 생성. 1건이면 대표 기사 값. 실패 시 대표 fallback.
전체 신뢰가중 집계importance_score · peak_importance_score · sentiment_score · risk_score · trust_score · grade_score · confidence_score · news_count여러 기사를 종합해야 정확한 점수들.
상위 3건top_evidence신뢰가중 상위 3개 헤드라인.
왜 event_type은 대표 1건, summary는 배치 종합? event_type은 백테스트 집계·disclosure_ref 매칭에 쓰는 기계용 단일 키라 하나여야 합니다. 반면 summary는 사람·LLM이 읽는 서술이라, 여러 사건이 섞였을 때 “인수·소송 포함, 종합 호재”처럼 전체를 종합하는 게 유용합니다. 그래서 유형은 대표, 서술은 배치 종합으로 나눴어요.
알아둘 트레이드오프 summary는 배치 종합이라 여러 사건을 함께 담지만, event_type은 대표 기사 1건 기준입니다. 그래서 숫자(집계)와 서술이 가리키는 사건이 어긋날 수 있어, 전략가는 summary만 보지 말고 sentiment_score·top_evidence를 함께 봐야 합니다.

주의 깊게 볼 컬럼 3개 — 상세

disclosure_ref — 뉴스 ↔ 공시 교차확인

뉴스 대표 event_type과 같은 종목·같은 날 공식 공시가 매칭되면 그 공시의 filing_no를, 없으면 None을 담습니다. 루머/팩트를 LLM으로 이진 판별하는 대신 “공식 문서가 실제로 존재하는가”로 사실성을 보강하는 장치예요. 매칭은 순수성 유지를 위해 스케줄러(집계 바깥)에서 주입합니다. 값이 있으면 to_prompt에 “✓ 공식공시 뒷받침”으로 표시됩니다.

peak_importance_score — 평균이 놓치는 특종

평균(importance_score)에 희석된 단독 강신호를 살리는 값. 단 신뢰검증 기사(확정 or 매체신뢰≥0.6 or 등급 ALLOW) 중에서만 최댓값을 뽑아, 저신뢰 매체의 과장이 가짜 peak를 세우는 걸 막습니다. max(importance_score, 신뢰기사 강도들)이라 항상 importance_score 이상. peak = importance_score면 “평균 이상으로 튀는 신뢰 신호 없음”을 뜻합니다.

top_evidence — 근거 헤드라인 3개

신뢰가중 상위 정확히 3개 기사 헤드라인(부족분은 None으로 채워 3칸 고정). 대표 기사 하나만으론 놓치는 다른 중요한 기사들의 흔적을 남겨, 전략가·감사가 “어떤 기사들이 이 신호를 만들었나”를 되짚게 합니다. 저장 시 접어서 TEXT로 보관.

confidence_score — 근거의 두께

기사별 무게(_news_conf)의 단순평균(가중 아님). 저신뢰 기사도 한 표를 온전히 행사해 값을 끌어내리므로 “근거가 얇다”를 정직하게 드러냅니다. 또 방향 점수 계산에서 호재 감쇠에 되먹임돼, 근거가 얇으면 호재를 더 세게 눌러 과신을 억제합니다. 전략가에겐 전달되지 않고 저장·계산용으로만 씁니다.

grade_score vs trust_score — 규칙 vs LLM

grade_score는 코드가 도메인 주소만 보고 매긴 3단계(ALLOW 1.0·GRAY 0.6·BLOCK 0.0)의 가중평균이고, trust_score는 LLM이 본문을 읽고 기사별로 매긴 source_trust의 신뢰가중 평균입니다. 둘을 독립으로 둔 이유는 이중 교차검증 — LLM이 잘 쓴 가짜뉴스에 속아도 도메인 등급이 잡고, 신뢰 매체가 이상한 기사를 내면 LLM 신뢰가 잡습니다.

hard_block · hard_block_reason — 즉시 정지

어느 기사·공시든 최종 권한이 BLOCK_ALL/BLOCK_BUY면 hard_block=1이 됩니다(점수 무관 즉시 매수 차단). 사유(hard_block_reason)는 전달 시 summary 맨 앞에 [차단: …]로 자동 삽입돼, 전략가가 “왜 막혔나”를 요약 첫 줄에서 바로 봅니다. 상폐·거래정지·감사인 교체 같은 하드리스크가 트리거예요.

8.계산 컬럼 공식 상세 + 예시

⚙️로 표시된 계산 컬럼의 공식입니다. 예시는 같은 3기사로 이어지며 값은 코드로 검증했습니다.

기사출처(등급)유형/방향강도확정?매체신뢰확신도위험
A1Reuters(ALLOW)product_deal/호재7✅0.9High1
A2Bloomberg(ALLOW)analyst_rating/호재5✅0.8Medium0
A3블로그(BLOCK*)product_deal/호재9❌0.3Low0

* A3(블로그)는 도메인상 BLOCK이라 실제로는 사전필터에서 drop됩니다. 아래 계산은 “저신뢰 기사가 집계에 들어와도 무게로 자동 무력화됨”을 보이는 설명용으로, A3가 걸러지지 않고 살아남았다고 가정한 경우입니다.

① _n10 — 0~10을 0~1로 기초

결과 = (0~10로 자름) ÷ 10, round(2)      # 7 → 0.7

② _news_conf — 기사 하나의 무게 기초

무게 = 확신도 × 확인계수 × 매체신뢰
   확신도 High.9/Med.6/Low.3 · 확인계수 확정1.0/소문0.5 · 매체신뢰 0~1
A1 = 0.9×1.0×0.9 = 0.81 · A2 = 0.6×1.0×0.8 = 0.48 · A3 = 0.3×0.5×0.3 = 0.045 · tw = 1.335
왜 곱(×)인가 곱셈은 “그리고(AND)” — 확실하고 확인됐고 믿을 매체여야 무게가 큼. 하나만 낮아도 전체가 급감(약한 고리). 더했다면 자극적 기사가 확신도만 높여 통과할 수 있음.

③ _senti_score — 방향(-1~+1)을 0~1로, 호재만 감쇠 기초

방향강도 = clamp(입력, -1, +1)
if 방향강도 > 0: 방향강도 ×= confidence     # 호재만 감쇠
결과 = 0.5 + 0.5 × 방향강도                  # 악재는 감쇠 없음
악재 -0.8 → 0.5+0.5×(-0.8) = 0.1 (또렷) · 호재 +0.8×conf0.45 → 0.36 → 0.68 (눌림)
왜 비대칭 근거 약한 낙관(호재)은 중립 쪽으로 눌러 과신 방지, 경고(악재)는 흐리지 않음 — “가장 보수적 채택”.

④ importance_score — 강도의 가중평균 Σ(값·w)/tw

importance_score = Σ(_n10(강도) × 무게) ÷ tw       # 부호 없음
(0.7×0.81 + 0.5×0.48 + 0.9×0.045) ÷ 1.335 = 0.8475 ÷ 1.335 = 0.63 # 산술평균이면 0.70 — A3(0.9)가 무게 0.045라 거의 무시돼 0.63으로 정직하게 낮음
왜 부호를 안 넣나 방향을 섞으면 큰 호재+큰 악재가 상쇄돼 0으로 붕괴. 강도(importance_score)와 방향(sentiment)은 다른 축.

⑤ peak_importance_score — 신뢰기사 중 최댓값 max

신뢰기사 = 확정 or 매체신뢰≥0.6 or 등급 ALLOW
peak = max(importance_score, 신뢰기사들의 강도…)
신뢰기사 A1(0.7)·A2(0.5), A3(0.9) 제외 → max(0.63, 0.7, 0.5) = 0.70
왜 신뢰기사만 저신뢰 과장이 가짜 peak(0.9)를 세우는 편향 차단. importance_score를 후보에 넣어 peak ≥ importance_score 보장. peak=importance_score면 “튀는 신뢰신호 없음”.

⑥ sentiment_score — 방향 집계(2층) Σ(signed·w)/tw + 매핑

# 1층: 방향 집계
signed = _n10(강도) × 부호               # -1~+1
if min(signed) ≤ -0.7: agg = min(signed)   # 강악재 우선
else: agg = Σ(signed×무게) ÷ tw
# 2층: _senti_score(agg, confidence)
signed [+0.7,+0.5,+0.9], 강악재 없음 → agg = 0.635 → _senti_score(0.635, 0.45) = 0.64 → positive
강악재 우선 가장 나쁜 기사가 -0.7 이하면 평균 대신 그 최악값 채택 — 치명 경고가 호재에 물타기되지 않게.

⑦ risk_score / trust_score / grade_score — 가중평균 3형제 Σ(값·w)/tw

risk_score = (0.1×0.81 + 0 + 0) ÷ 1.335 = 0.06 # A1만 위험 trust_score = (0.9×0.81 + 0.8×0.48 + 0.3×0.045) ÷ 1.335 = 0.84 # 기사별 source_trust(LLM 매체신뢰)의 가중평균 grade_score = (1.0×0.81 + 1.0×0.48 + 0.6×0.045) ÷ 1.335 = 0.99 # 코드 도메인등급
trust_score(0.84) vs grade_score(0.99) LLM은 내용 읽고 기사별 source_trust를 신중히(로이터도 0.9), 코드는 도메인만 보고 ALLOW=1.0. LLM vs 규칙 이중 교차검증 — 하나가 속아도 다른 하나가 잡음.

⑧ confidence_score — 무게의 단순평균 Σw/n

confidence_score = Σ(무게) ÷ 기사 수(n)        # 가중 아님! raw_weights 사용
(0.81+0.48+0.045) ÷ 3 = 0.45
왜 단순평균(importance_score와 반대) confidence_score는 “근거가 두꺼운지”를 정직하게 드러내야 함 → 저신뢰 기사도 한 표를 온전히 행사해 끌어내림. 가중이면 약한 기사가 무시돼 가짜로 높아짐. 되먹임: confidence_score는 ③의 호재 감쇠에 다시 들어가 약한 근거의 낙관을 억제.

강도와 방향이 갈라지는 순간 — 두 가지 대비 예제

모두 호재였던 앞 예제와 달리, 호재·악재가 섞이면 importance_score(강도)와 sentiment_score(방향)가 확 벌어집니다. 왜 두 축을 분리했는지 여기서 분명해져요.

대비 예제 A · 혼조 (큰 호재 + 큰 악재)

둘 다 확정·고신뢰이고 강도가 같은(6/10) 호재 1건 + 악재 1건.

무게 = [0.81, 0.81], tw = 1.62, 강도 mags = [0.6, 0.6] importance_score = (0.6×0.81 + 0.6×0.81) ÷ 1.62 = 0.6 # 강도는 그대로 높음 방향 signed = [+0.6, -0.6] → 가중평균 agg = 0 → sentiment_score = 0.5 # 방향은 상쇄 호재무게·악재무게 각 50% ≥ 30% → sentiment = mixed
핵심 만약 강도에 방향을 섞었다면 importance_score도 0으로 붕괴해 “둘 다 큰일인데 중요도 0”이 됐을 것. 강도(0.6)는 유지하면서 방향만 중립(0.5)으로 나오는 게 두 축 분리의 목적입니다. 라벨은 mixed로 “호·악재 공존”을 정직하게 표시.

대비 예제 B · 강한 악재 우선 (호재 + 상폐 악재)

실적 호재(강도 5) + 상장폐지 악재(강도 9), 둘 다 확정·고신뢰.

무게 = [0.81, 0.81], tw = 1.62 importance_score = (0.5×0.81 + 0.9×0.81) ÷ 1.62 = 0.7 방향 signed = [+0.5, -0.9] → 가장 나쁜 게 -0.9 ≤ -0.7 → 강악재 우선 발동 agg = min(signed) = -0.9 (평균 아님) → sentiment_score = 0.05 → sentiment = negative
핵심 호재가 있어도 치명적 악재 1건이 방향을 지배합니다. 평균냈다면 호재가 악재를 물타기해 “약한 악재”쯤으로 흐려졌겠지만, 강악재 우선 규칙이 0.05(강한 악재)로 또렷하게 남겨요 — 안전 우선 사상.

10.헷갈리는 축 3쌍

쌍1 · importance_score ↔ sentiment_score 강도 vs 방향. 큰 악재는 importance_score 높고 sentiment_score 낮음. 독립 축이라 못 합침.
쌍2 · confidence_score ↔ trust_score ↔ grade_score 근거 두께(무게 단순평균) / LLM 매체신뢰(가중) / 코드 도메인등급(가중). LLM·규칙 이중 교차검증이라 일부러 분리.
쌍3 · importance_score ↔ peak_importance_score 평균 vs (신뢰기사 중) 최고. peak ≥ importance_score 항상 성립. peak가 더 크면 “평균에 희석된 강신호 1건 있음”.

점수 구간 읽는 법

0~1 점수는 감이 안 올 수 있어 대략의 해석 구간을 둡니다(엄격한 임계값이 아니라 읽기 가이드).

구간sentiment_scoreimportance_score / risk_scoretrust_score / grade_score
0.0~0.3강한 악재낮음(사소)낮음(의심)
0.3~0.5약한 악재중하중하
0.5중립 (또는 mixed)——
0.5~0.7약한 호재중상중상
0.7~1.0강한 호재높음(중대)높음(신뢰)

참고: sentiment 라벨 규칙은 강악재(≤ −0.7 방향값) 1건이면 negative, 호재무게·악재무게가 각 30% 이상이면 mixed, 그 외엔 점수 기준(>.5 positive / <.5 negative / =.5 neutral)입니다. grade_score는 사실상 세 값(ALLOW 1.0 · GRAY 0.6 · BLOCK 0.0)의 가중평균이라 대부분 0.6~1.0 사이에 분포합니다.

전달 스코어를 어떻게 조합할까 — 합리적 가중치 제안

먼저 분명히 최종 매수 결정과 가중치는 Strategist(이은미)의 몫입니다. 아래는 “이 스코어들을 하나로 합친다면 이렇게 시작하는 게 합리적”이라는 출발점 제안이며, 실제 값은 백테스트로 튜닝해야 합니다. 정보분석 파트가 강제하는 규칙이 아니라 참고안이에요.

스코어들은 성격이 달라 전부 더하면(단순 가중합) 안 됩니다. 세 종류로 나눠 다뤄야 합리적이에요.

역할필드어떻게 쓰나

가중합(방향·강도·위험)sentiment_score · importance_score · peak_importance_score · risk_score아래 가중치로 합성.

제안 합성식 (0~1 확신도 C, 매수/매도 방향은 sentiment 부호)

# 방향 강도(0~1): 0.5(중립)에서 얼마나 벗어났나
D = |sentiment_score − 0.5| × 2        # 부호: >0.5 매수 / <0.5 매도
# 강도(확신): 평균 위주 + 단독 강신호 일부
M = 0.6×importance_score + 0.4×peak_importance_score
# 가중합 → 위험 감점
Raw = 0.50×D + 0.35×M − 0.15×risk_score
# 신뢰로 전체 할인 + 게이트
C = clamp(Raw, 0, 1) × trust_score ;  hard_block이면 C = 0
구성제안 가중치근거
방향(D, sentiment_score)0.50 (최대)매수냐 매도냐 자체 — 가장 중요. sentiment_score는 이미 호재 감쇠·강악재 우선이 반영돼 있어 신뢰할 만함.
강도(M, importance_score·peak)0.35포지션 크기(확신). 평균(importance_score) 0.6 + 단독 강신호(peak) 0.4로 섞어 큰 단발 신호도 반영.
위험(risk_score)−0.15 (감점)하방 위험은 사이즈를 줄이는 방향. 과하게 크면 좋은 호재도 다 죽으니 0.15 정도.
신뢰(trust_score)× 곱셈못 믿을 신호는 강도와 무관하게 통째로 할인 — “믿을수록 크게”를 결정 단계에서도 유지.
예) 뉴스 예시1(sentiment 0.64·imp 0.63·peak 0.70·risk 0.06·trust 0.84) D = |0.64−0.5|×2 = 0.28 · M = 0.6×0.63 + 0.4×0.70 = 0.66 Raw = 0.50×0.28 + 0.35×0.66 − 0.15×0.06 = 0.14 + 0.231 − 0.009 = 0.36 C = 0.36 × 0.84(trust) = 0.30 → 방향 매수(0.64>0.5), 확신 中하 → 소액 진입 수준
읽는 법 C가 낮게(0.30) 나온 건 sentiment_score가 0.64로 중립(0.5)에서 크게 안 벗어났기 때문이에요(확신도 감쇠 탓). “강한 호재”가 아니라 “완만한 호재”라 사이즈를 크게 잡지 않는 게 합리적이라는 신호입니다. 공시는 peak_importance_score·trust_score가 없으니 M=importance_score, 곱셈 배수 없이 같은 틀을 단순화해 쓰면 됩니다.

FAQ

Q. 컬럼 29개가 너무 많지 않나? 전략가가 읽는 건 15개뿐. 나머지는 감사·근거 창고라 “많다”의 체감 주체가 없음. 각 점수에 붙는 *_score_reason도 판단 근거를 남기는 감사용이라 부담이 아닙니다.
Q. 공시는 왜 trust_score·peak가 없나? 출처가 SEC라 매체신뢰 불필요, 1건이라 집계·최댓값 개념도 없음.
Q. summary가 1건일 땐? 배치 종합은 2건 이상일 때만. 1건이면 그 기사 요약이 곧 전체라 대표 값을 그대로 씀. LLM 합성 실패 시에도 대표 fallback.
Q. 소셜은 정말 다 막히나? 네. 소셜·포럼·블로그·미등록 도메인은 BLOCK → LLM 호출 전 drop. GRAY는 등록된 2급 언론사만.
Q. confidence_score·trust_score·grade_score는 결국 다 “신뢰” 아닌가? 아니요. 근거 두께(무게 단순평균) / LLM이 본 매체 신뢰(가중) / 코드 도메인 등급(가중)으로 서로 다릅니다. 특히 뒤 둘은 LLM·규칙을 독립으로 둔 이중 교차검증이라 일부러 안 합칩니다.
Q. ALLOW면 “사도 된다”는 뜻인가? 아닙니다. ALLOW/GRAY/BLOCK은 출처 신뢰 등급일 뿐이고, 매수 판단은 Strategist 몫입니다. 우리 쪽 차단은 hard_block이 담당합니다.
Q. sentiment_score 0.5는 “호재 절반”? 아니요, 중립입니다(방향 없음). 강도는 importance_score가 따로 표현합니다.
Q. 저신뢰 기사를 왜 아예 안 버리나? 완전히 버리는 대신 무게를 아주 작게 줍니다. 그래야 confidence_score가 “근거 얇음”을 정직하게 낮게 반영해요. (명백한 노이즈·BLOCK 도메인은 LLM 전에 이미 제거.)
Q. event_type이 하나뿐인데 여러 사건이 섞이면? event_type은 백테스트·매칭용 단일 키라 대표 1건 기준이고, 섞인 사건들은 summary(배치 종합)가 문장으로 풀어줍니다.
Q. 같은 입력인데 결과가 매번 달라질 수 있나? 아니요. LLM 온도(temperature)를 0으로 고정해 같은 입력이면 항상 같은 출력(결정론)이 나오게 했습니다. 정보분석은 ‘창작’이 아니라 ‘판정’이라 무작위성이 오히려 노이즈이기 때문이에요.
Q. 시각은 왜 UTC로 저장하나? 백테스트의 look-ahead 방지와 미국장 정합을 위해 저장은 UTC로 통일합니다. 사람이 볼 때만 KST로 변환해 표시해요(저장≠표시).

13.설계 철학 — 모든 공식이 향하는 두 원리

앞의 복잡해 보이는 공식들은 사실 두 개의 철학으로 수렴합니다. 이 둘만 잡으면 각 장치가 왜 그렇게 생겼는지 전부 설명돼요.

철학 ① · “믿을수록 크게” (신뢰가중) 모든 집계는 단순 평균이 아니라 신뢰도로 가중한 평균입니다. 믿을 만한 기사는 큰 목소리, 못 믿을 기사는 작은 목소리. 그래서 시끄러운 루머 몇 개가 진짜 신호를 오염시키지 못합니다.
철학 ② · “가장 보수적으로” (겁쟁이처럼) 확실하지 않은 좋은 소식은 의심하고, 나쁜 소식(경고)은 또렷하게 남깁니다. 여러 판단이 갈리면 가장 제한적인 쪽을 택하고, 확인 전엔 매수를 보류합니다. 좋은 기회를 몇 개 놓치더라도 크게 잃지 않는 것이 우선.

15.백테스트 & 수익률 측정

“이 신호가 정말 맞았나”를 며칠 뒤 실제 주가로 채점합니다. 저장 스냅샷이 감사·백테스트 창고인 이유예요.

  • 기준일(day0) = 공시는 filed_at(없으면 analyzed_at), 뉴스는 published_at. look-ahead(미래 정보 참조) 방지.
  • 전방 수익률 = day0 대비 +1 / +3 / +5 거래일 종가 수익률(Yahoo 일봉). return_1d/3d/5d 컬럼에 채움.
  • 적중 판정(outcome) = 우리가 말한 방향(호재/악재)이 실제 주가 방향과 맞으면 hit, 틀리면 miss.
  • 리포트 = 이벤트/센티먼트/출처별 건수·평균수익·적중률 집계.
컬럼뜻
return_1d / 3d / 5dday0 대비 +1/+3/+5 거래일 종가 수익률(%)
outcome센티먼트 방향 적중 hit / miss
예) 호재 신호(sentiment_score 0.7) → 3일 뒤 +2.1% → 방향 맞음 → outcome = hit 악재 신호(0.2) → 3일 뒤 +1.5%(올랐음) → 방향 틀림 → outcome = miss

측정까지만, 자동 튜닝은 안 함 표본이 아직 적은데 섣불리 규칙(가중치)을 바꾸면 과거에만 잘 맞는 이상한 규칙(과적합)이 됩니다. 그래서 현재는 측정·리포트까지만 하고 가중치 자동조정은 하지 않습니다(표본이 쌓이면 팀 검토 후 튜닝).
