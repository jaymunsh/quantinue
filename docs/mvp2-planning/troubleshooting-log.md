## T1. 모든 종목이 NVDA 공시로 채점되고 있었다 🔴

**증상**: AAPL을 분석해도 `tb_disclosure_signal`에 NVIDIA의 accession(`0001310264-…`)이 기록된다.

**원인**: `role_05_disclosure_analysis/service.py:51`이 CIK를 상수로 박아 조회했다.
```python
filings = await self.market_data.sec_submissions("1045810", ...)  # = NVDA
```

**왜 위험한가**: `disclosure_score`는 ① role_07의 4표 중 1표 ② 강악재 차단 게이트(`hard_negative_max`)의 입력이다. 즉 **M4에서 "완료"로 기록한 방어선이 거짓 입력 위에서 돌고 있었다.**

**조치**: SEC `company_tickers.json`으로 티커→CIK 해석(프로세스당 1회 캐시). 미해결 시 **기권**(§T2).

**교훈**: 게이트를 완성했다고 기록하기 전에 **그 게이트의 입력이 진짜인지** 확인한다. 게이트 로직의 테스트는 입력의 진실성을 검증하지 않는다.

## T2. 공시가 없으면 파이프라인이 죽었다 🔴

**증상**: T1을 고치자 "공시 없는 종목"이 실제로 생기면서 런이 실패.

**원인**: `role_07/service.py:36`의 `require_value(context.disclosure_score)`. 그전까지는 항상 NVDA 공시가 있어서 드러나지 않았다 — **T1이 T2를 가리고 있었다.**

**조치**: 부재를 **투표 기권**으로 표현. 0점이 아니다.

**왜 0점이 아닌가**: 0점은 중립이 아니라 **강한 악재**로 읽혀 `hard_negative_max`(0.15)에 걸린다. 공시 없는 종목의 매수가 전부 차단됐을 것이다.

**교훈**: 🔑 **"모르는 상태"를 표현할 자리를 먼저 만든다.** 이 규칙이 M4 전체를 관통했다 — 공시 부재(T1)·소유권 신고서(T11)·저신뢰 뉴스가 전부 같은 함정이었다. 없는 값을 0으로 채우면 그 0이 의미를 갖는 곳에서 오작동한다.

## T3. 거래정지 종목을 만나면 사이클 전체가 실패했다 🔴

**증상**: halted 가드가 주문을 생략하면 role_11에서 `MissingStageDataError: component 11 requires order`.

**원인**: 생략 시 `order_skipped_reason`만 세우고 `quantity`를 그대로 뒀다. role_11(`processor.py:77`)은 `not context.quantity`로 "주문 없음"을 판정하므로, 수량이 남아 있으면 주문을 요구한다.

**조치**: 생략 시 `quantity=0`도 함께 내린다.

**어떻게 발견했나**: 유닛 테스트는 role_10만 검증해서 통과했다. **가드를 강제로 발동시키는 E2E**를 짜면서 드러났다.

**교훈**: 가드가 상태를 바꾸면 **그 상태를 읽는 하류 전부**를 확인한다. 역할 하나만 보는 테스트는 역할 경계를 넘는 계약을 못 잡는다.

## T4. 갭 가드가 파이프라인을 죽였다 🔴

**증상**: 유닛 646건 전부 green인데 통합 테스트에서 `exchange_calendars.errors.DateOutOfBounds`.

**원인**: 거래소 캘린더는 유한 구간(약 1년)만 안다. 통합 테스트의 cycle_ts가 2035년이라 `is_trading_day()`가 예외를 던졌고, 갭 가드가 그걸 잡지 않아 런이 죽었다.

**조치**: 세션 판정 불가 시 측정을 생략한다(가드가 파이프라인을 죽이는 쪽이 더 나쁘다).

**교훈**: **유닛 green이 통합 green을 뜻하지 않는다.** 외부 라이브러리의 정의역(캘린더 범위, 스키마 제약)은 mock이 알려주지 않는다.

## T5. SEC와 NASDAQ의 User-Agent 요구가 정면 충돌했다 🟠

**증상 1**: 새로 붙인 SEC ticker-map fetch가 **403**.
**증상 2**: 그걸 전역 UA 변경으로 고쳤더니 이번엔 **NASDAQ 스크리너가 연결을 끊어** role_01이 전멸.

**원인**: 실측 결과 두 상류의 요구가 반대다.

| 엔드포인트 | 연락처 없는 UA | 연락처 있는 UA |
|---|---|---|
| `www.sec.gov` | **403** | 200 |
| `data.sec.gov` | 200 | 200 |
| `api.nasdaq.com` | 200 | **연결 리셋** |

SEC 공정접근 정책은 연락 가능한 주소가 담긴 UA를 요구한다(`data.sec.gov`는 강제하지 않아 그간 미준수인 줄도 몰랐다). NASDAQ은 그 형식을 거부한다.

**조치**: 연락처 UA를 **SEC 요청에만 헤더로** 부착. 공용 클라이언트는 제품 토큰 유지. `QUANTINUE_HTTP_USER_AGENT`로 재정의 가능.

**교훈**: 🔑 **전역 설정으로 외부 규약을 맞추려는 시도는 소스가 둘 이상이면 깨진다. 소스별 정책은 소스별로.** 그리고 **mock transport는 상대 서버의 정책을 흉내내지 않는다** — 새 외부 엔드포인트는 반드시 실호출로 한 번 찔러본다.

> 부수: NASDAQ에서 HTTP/2 `StreamReset`이 간헐 발생. 재시도로 통과 — 일시적 상류 문제로 판단, 코드 변경 없음.

## T6. 게이트가 영원히 걸리지 않았다 (100배 불일치) 🟠

**증상**: `late_entry_max`를 배선해도 발동하지 않는다.

**원인**: `role_02/service.py:128`이 `ret_5d`를 **퍼센트**로 계산(`*100`)하는데, `profiles[].late_entry_max`는 **분수**(0.15)다. 22% 상승이 `22.0`으로 도착해 `0.15`와 비교된다 — 항상 초과하거나, 반대 방향 비교면 절대 걸리지 않는다.

**조치**: 역할 경계에서 정규화(`ret_5d / 100`), 변환을 한 곳에 고정.

**교훈**: 같은 물리량의 **단위를 계약에 명시**한다. 필드명만으로는 퍼센트인지 분수인지 알 수 없다.

## T7. 시스템이 문서보다 10포인트 느슨하게 승인하고 있었다 🟠

**증상**: 설계 문서·playbook은 크리틱 승인 문턱 0.70인데 실제로는 0.60으로 동작.

**원인**: 값이 두 곳에 있었다 — `mvp2.gates.critic_approval`(0.70, M2 설계값)과 낡은 `mvp.thresholds.critic_approval_score`(0.60). `factory.py:107`이 **낡은 쪽**을 배선했고, 새 값은 죽어 있었다.

**조치**: `gates`를 유일 소유자로. 낡은 키를 yaml·모델·테스트에서 삭제.

**교훈**: 마이그레이션에서 **새 설정을 추가할 때 낡은 것을 같이 지운다.** 둘이 공존하면 조용한 쪽이 이긴다.

## T8. 첫 주문이 종목당 $250,000 규모였다 🟠

**증상**: 무장 직전 점검에서 발견. 의도한 주문 크기가 아니었다.

**원인**: `max_app_order_exposure_usd`가 **두 가지 일**을 한다 — ① 사이징의 `equity`(`role_09/service.py:101`) ② 누적 노출 천장(예약 로직). `.env` 실값이 $1,000,000이었다.
```
포지션당 = min(1,000,000 × 0.04/0.15, 1,000,000 × 0.25) = $250,000
```

**조치**: 즉시 `.env`를 90,000으로(임시). 이후 M6-2에서 **자본은 계좌에서, 제약은 프로필에서** 오도록 분리 — 근본 해소.

**교훈**: 하나의 값이 두 의미를 가지면 **어느 쪽도 독립적으로 조정할 수 없다.** 이 경우 포지션 크기가 항상 자본의 정확히 25%로 고정되어 현금 비율·비중을 따로 정할 방법이 없었다.

## T9. 방어선이 몇 번 발동했는지 셀 수 없었다 🟡

**증상**: `premarket_gap_max` 문턱을 "1~2주 관측 후 보정"하기로 했는데, 그 관측이 불가능.

**원인**: role_09의 결정이 **어디에도 저장되지 않았다.** 주문이 생긴 경우만 `tb_order`에 남고, 가드가 막은 경우는 `pipeline_checkpoints.payload` JSONB 안의 요약 문자열이 전부였다. 게다가 config 주석이 `tb_order_plan`에 기록된다고 적고 있었는데 **그런 테이블이 존재하지 않았다.**

**조치**: `tb_order_plan` 신설(집행/보류·사유·수량). CHECK로 집행↔보류 상호배타 고정.

**교훈**: **관측 계획이 있는 문턱은 관측 수단을 같은 마일스톤에 만든다.** "나중에 로그 보고 조정"은 로그가 조회 가능할 때만 계획이다.

## T10. 만장일치에서 INSERT가 깨졌다 🟡

**증상**: `signal_consensus` 실계산을 켜자 4표 동의 시 제약 위반.

**원인**: `CHECK (signal_consensus BETWEEN 0 AND 3)`인데 투표원은 4개(technical·disclosure·news·model). 그전까지 리터럴 0만 써서 드러나지 않았다.

**조치**: `0~4`로 확장(schema + migration + 계약 기대치 동시).

**교훈**: 하드코딩된 값이 제약을 **가려준다.** 실계산을 켜는 순간 제약이 처음으로 시험된다.

---

## T11. 소유권 신고서가 실질 공시를 밀어냈다 🟡

**증상**: 실행 DB 확인 중 `tb_disclosure_signal`에 `xslF345X06/wk-form3_….xml`이 공시 시그널로 기록된 것을 발견.

**원인**: `filings[0]`은 **'가장 최근'일 뿐**이고 종류를 가리지 않는다. 대형주는 인사이더 Form 3/4를 상시 제출하므로 정작 채점하려던 8-K를 일상적으로 밀어낸다. 게다가 `form_type`이 `"8-K"`로 하드코딩(`service.py:138`)돼 **무엇이 오든 실질 이벤트로 라벨링**됐다.

**조치**: config `mvp2.disclosure.llm_bypass_forms`(3/4/4-A/5) 제외 후 실질 공시를 채점. 전부 소유권 신고서면 기권. `form_type`을 실제 값으로 기록.

**미해소**: `filings[0]`이 **날짜 무관**이라 3개월 전 공시가 오늘의 시그널이 될 수 있다. 신선도 창 필요(⏳ 보완 목록).

## T12. 감사 보고 자체가 세 번 틀렸다 🟠 (메타)

**증상**: 유령 설정 감사 결과를 보고했는데, 그중 셋이 사실과 달랐다.

| 오보 | 실제 |
|---|---|
| "픽 50개 전부가 LLM을 탄다" | **다종목 팬아웃이 애초에 없다.** 모든 런이 단일 종목 |
| "리스크 한도가 하나도 없다" | 작동하는 한도가 둘 있었고, 문제는 그 **값**이었다(T8) |
| "5건 × 25% = 125% 과다배분" | 예약 함수 안 한도가 **둘**인데 하나만 보고 계산. 실제로는 4건에서 멈춰 **100%** |

**원인**: `grep`으로 "소비자 없음"을 확인한 뒤 **그 결과를 추정으로 확장**했다. 세 번 다 같은 실수다.

**교훈**: 🔑 **정적 검색은 후보를 줄 뿐 결론을 주지 않는다.** "소비자 없음"은 기계적으로 판정되지만 **"그래서 무엇이 위험한가"는 호출 경로를 끝까지 읽어야** 알 수 있다. 유령 목록의 각 항목은 결과를 코드로 확인하기 전에 보고하지 않는다.

## T13. 팬아웃을 붙이자 전역 천장이 계좌별 사이징과 충돌했다 🟡

**증상**: 계좌별 사이징을 켜자 $100,000 계좌 두 개가 전부 수량 0. $5,000 계좌만 통과.

**원인**: 일일 주문 예약이 **건수 한도와 금액 한도 둘**을 함께 본다(§T12에서 배운 그 구조). 금액 한도가 **앱 전역 단일 값**이라, 자본이 다른 계좌들이 천장 하나를 나눠 쓰게 된다 — 큰 계좌가 먼저 소진하고 나머지는 거부된다.

**조치**: 노출 천장을 **계좌 자본**으로. 계좌별로 의미 있는 상한이 된다("이 계좌의 계획 주문 총액은 이 계좌 자본을 넘지 않는다").

**교훈**: 단일 계좌 전제로 만든 전역 한도는 **다계좌를 도입하는 순간 의미가 뒤집힌다.** 한도의 적용 단위가 도메인 단위(계좌)와 일치하는지 확인한다.

---

## T14. 다계좌 집행이 실 브로커에 7배 주문을 낼 뻔했다 🔴 (예방)

**상황**: 데모 5 + 테스트 2 = 내부 7계좌를 순회해 집행하도록 만들던 중.

**위험**: 내부 계좌는 7개지만 **실 브로커 뒤의 외부 계좌는 하나**다(Alpaca 키 1쌍). 순회를 그대로 실 브로커에 적용하면 같은 종목에 7건이 제출되어 **의도한 포지션의 7배**가 한 계좌에 쌓인다. 페이퍼라 손실은 가짜지만 T+5 학습 데이터는 진짜다.

**조치**: 팬아웃을 **시뮬 브로커 한정**으로 계약화. 실 브로커는 대표 계좌 1건만 제출하며, 이를 테스트로 고정("a real broker never fans out").

**교훈**: **내부 도메인 단위와 외부 시스템 단위가 1:1이라고 가정하지 않는다.** 계좌·주문·포지션 같은 개념은 우리 원장과 브로커에서 서로 다른 기수(cardinality)를 가질 수 있다. 팬아웃을 도입할 때 "이 순회가 외부에 몇 번 닿는가"를 먼저 센다.

## 검증 계층에 대한 총평

오늘 발견된 결함을 **무엇이 잡았는가**로 분류하면:

| 잡은 계층 | 결함 |
|---|---|
| 유닛 테스트 | T6(단위 불일치)·T10(제약 범위) |
| **통합 테스트** | T4(캘린더 정의역) |
| **강제 발동 E2E** | T3(하류 계약 파손) |
| **실행(드라이런)** | T1·T5·T8·T11 |
| **코드 정독** | T7·T9·T12 |

**실행에서만 드러난 것이 가장 많다.** 그리고 그것들이 대체로 가장 위험했다 — 판단 오염(T1·T11), 수집 중단(T5), 규모 오류(T8). mock은 우리가 만든 세계라서 **우리가 안 만든 현실은 보여주지 않는다.**
