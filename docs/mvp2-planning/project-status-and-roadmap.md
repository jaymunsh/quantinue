## 0. 지금 상태 한 장 요약

Quantinue MVP-2는 **실제 시장 데이터를 수집하고, LLM 전략가와 비평가가 판단한
결과를 모의 계좌에 자동 집행하며, 관리자와 사용자 화면에서 그 전 과정을
추적하는 자동매매 관측 시스템**이다. 실계좌 주문은 목표가 아니며 체결은
`MockBroker`가 맡는다.

현재 HEAD에는 일일 파이프라인과 운영 감시 구현이 있다. 장중 후보 설정은 가격
폴링만 켜고, LLM 재판단과 웹소켓 스트림은 끈 상태다. 기존 8020이 이 후보를
재로딩했는지는 별도 런타임·원장 증거가 필요하다.
공시·뉴스가 장중 새로 들어왔을 때 관련 종목만 재판단하는 경로는 다음 개발이다.

| 구분 | 상태 | 실제 의미 |
|---|---|---|
| 일일 JOB 체인 | **운영 중** | 뉴욕 거래 슬롯마다 수집→선별→LLM→청산→배분 |
| OpenAI 판단 | **운영 중** | `gpt-5.4-mini`, 사용량 원장과 `$3/일` 상한 적용 |
| 장중 가격 폴링 | **설정 후보 ON** | HEAD는 정규장 1분, 실제 8020 적용은 확인 대기 |
| 장중 브래킷 방어 | **구현 완료** | 손절·익절 도달 시 같은 tick에서 모의 청산 |
| ±5%·정기 LLM 재판단 | **구현 완료·비활성** | `rejudge=false`, 운영 검증 뒤 활성화 예정 |
| 보유 종목 웹소켓 | **구현 완료·비활성** | `stream=false`, 30종목 한도 실측 완료 |
| 장중 공시·뉴스 증분 수집 | **미구현** | 현재 `disclosures/news/news_wire`는 하루 1회 |
| 외부 dead-man 감시 | **구현·설정 완료** | 실제 송신 상태는 8020 런타임과 외부 원장으로 확인 |
| AWS 이전 | **미착수** | 로컬 검증 후 선택, 최소안은 Lightsail 2GB |

**따라서 “개발이 다 끝났다”는 상태는 아니다.** 가격 중심의 장중 기반은
완성됐고, 제품 의도에 필요한 마지막 큰 조각은 **장중 신규 공시·뉴스 → 관련
종목 재판단** 연결이다.

## 1. 프로젝트가 지키는 원칙

| 원칙 | 현재 확정 |
|---|---|
| 체결 | 실제 주문이 아닌 로컬 모의 체결이 최종 상태 |
| 시세·근거 | Alpaca, SEC, FRED, RSS 등 실제 외부 데이터 사용 |
| 판단 | LLM 전략가 제안 뒤 critic 승인·기각을 거쳐야 함 |
| 새 종목 발굴 | 거래일마다 전체 파이프라인 1회 |
| 보유·당일 후보 대응 | 장중 가격과 사건에 반응하도록 별도 감시 층 사용 |
| 안전 | 손절·익절은 LLM과 분리된 결정론 경로 |
| 비용 | 모든 유료 LLM 호출을 기록하고 일일 상한에서 fail-closed |
| 시간 | DB 원장 UTC, 화면 KST, 거래 슬롯은 뉴욕 날짜 |
| 설정 | 주기·문턱·한도·요율은 `config/pipeline.yaml` 한곳이 소유 |
| 운영 | 8020만 작업 소유, 개발 8021은 web-only + mock |

전체 JOB 체인을 장중 여러 번 반복하지 않는다. 변하지 않는 일봉·전체
스크리닝은 하루 1회 두고, 가격·공시·뉴스처럼 장중 변하는 것만 가벼운 별도
루프로 처리한다.

### 2-1. 일일 등록 JOB 14종

러너는 60초마다 due 여부를 보지만, `tb_job_run(job_name, slot_date)`가 같은
뉴욕 슬롯의 중복 실행을 막는다. `universe`는 7일, 나머지는 거래 슬롯당
1회다. 조건부 잡이나 due가 아닌 잡은 그날 실행 행이 없을 수 있으므로
“등록 14종”과 “그날 성공 13행”은 모순이 아니다.

| 순서 | JOB | 주기 | 책임 |
|---:|---|---:|---|
| 1 | `universe` | 7일 | NASDAQ 보통주+보유 종목 유니버스 |

### 2-2. `news_wire`란 무엇인가

`news`는 Alpaca가 제공하는 Benzinga 계열 기사이고, `news_wire`는 기업·기관이
직접 발행하는 RSS/Atom 보도자료를 모으는 보조 수집기다. Alpaca 뉴스에 늦거나
빠진 발표를 보완한다. 별도 전략가는 아니며 앞으로 장중화할 때 일반 뉴스와
중복 제거한 뒤 같은 사건 라우터로 보낸다.

## 5. 아직 남은 것

### 5-1. 최우선: 장중 신규 공시·뉴스 재판단

현재 공시·뉴스·와이어는 일일 체인에서 한 번만 돈다. 목표는 전체 체인을 여러
번 돌리는 것이 아니라 다음 세 루프를 분리하는 것이다.

| 수집 | 목표 주기 | 이후 동작 |
|---|---:|---|
| SEC 공시 | 30분 | 새 중요 공시의 관련 종목만 재판단 |
| Alpaca/Benzinga 뉴스 | 15분 | 새 중요 뉴스의 관련 종목만 재판단 |
| RSS/Atom 와이어 | 10분 | 일반 뉴스와 중복 제거 후 같은 경로 |

필요한 구성:

1. source별 cursor/watermark와 event unique key
2. 재기동·네트워크 실패 뒤 안전한 재개
3. 출처·중요도·티커의 결정론 필터
4. 보유 종목+당일 후보와의 교집합
5. 같은 `event+ticker+persona` 중복 LLM 방지
6. 기존 30분 cooldown·20% 매도 예약·`$3` 상한 공유
7. 판단이 바뀐 종목만 기존 청산/배분 경로로 표적 실행
8. 관제실에서 정상 무사건과 수집 중단을 구분

### 5-2. 운영 활성화 관문

코드가 있다는 사실만으로 켜지 않는다.

1. poll-only 정규장 ready 기록 확인
2. 깨끗한 8020-only 일일 슬롯과 OpenAI usage·retry 대조
3. `rejudge=true`로 올려 sweep·±5% 판단을 제한적으로 실측
4. 그 뒤 `stream=true`로 보유 종목만 웹소켓 활성화
5. 실제 정규장 1회 기능 확인, 다음 거래일 안정성 확인

### 5-3. 관측으로만 답할 항목

- 인사이더 투표가 실제로 서는 빈도와 문턱 조정 여부
- `premarket_gap_max=0.03`의 실제 갭 분포 적합성
- LLM 재판단 횟수·반전율·cooldown 억제 건수
- 20% 매도 예약 예산이 실제로 필요한지
- 무인 연속 완주와 외부 heartbeat 누락 이력

### 5-4. 나중에 정리할 기술 부채

- 구 `pipeline_runs` 계열 스키마: 관측 데이터 보존 판단 뒤 제거
- broker의 `is_tradable()` 계약은 있으나 allocation 제출 직전 연결은 아직 없음:
  장중 신규 매수를 활성화하기 전에 fail-closed 가드로 연결
- 소비자 없는 원장 컬럼: 억지로 채우지 않고 실제 소비자가 생길 때 재검토
- 상세 사건 탐색 UI: 핵심 장중 경로가 안정된 뒤
- RDS·로드밸런서·관리자 주기 조정 UI: 현재 규모에서는 만들지 않음

## 6. 앞으로의 실행 계획

### Phase A — 장중 사건 경로 2일 압축 개발

| 날 | 구현 | 완료 증거 |
|---|---|---|
| 1일차 | OpenAI Standard 요율 정정, cursor·중복 방지, 공시 30분·뉴스 15분·와이어 10분 증분 수집 | 중복 입력·재기동·일시 실패에도 사건이 정확히 한 번 저장 |
| 2일차 | 사건 라우터, 기존 전략가/critic 연결, cooldown·예산·표적 주문, 핵심 로그·Telegram | mock 압축 주기에서 무사건 0콜, 관련 사건 1회 판단, 중복 주문 0 |

2일에 포함하지 않는 것: 상세 사건 탐색 UI, 웹소켓 추가 개발, AWS 이전.

### Phase B — 실제 장중 검증

1. 8021+mock으로 짧은 주기를 사용해 반복·장애·재기동 검증
2. 8020 적용 전 DB 백업과 migration 2회 멱등 확인
3. 정규장에서 실제 신규/무사건 폴링과 LLM 원장 대조
4. 다음 거래일 중복 호출·중복 주문이 없는지 재확인
5. 결과를 `dev-handoff.md`, `open-items.md`, runbook에 반영

코드 완성은 2일, 운영 확정은 이후 1~2 거래일 관측이다.

### Phase C — 기존 장중 기능 단계 활성화

1. `rejudge=true`, `stream=false`
2. 하루 3회 sweep과 ±5% 사건 비용·판단 품질 확인
3. 이상 없으면 `stream=true`
4. 보유 종목 스트림, 당일 후보 폴링 fallback 확인
5. rollback은 설정을 끄는 것으로 끝내고 이미 쌓인 원장은 보존

### Phase D — AWS 이전 (선택)

권고 최소안은 Lightsail Small 2GB 한 대에 앱+PostgreSQL을 함께 두는 것이다.

| 항목 | 예상 |
|---|---:|
| 인프라 | 월 `$13~15`, 10일 약 `$4.3~5.0` |
| 구축·이전 | 1.5~2.5 개발일 |
| 포함 | public IPv4, 60GB SSD, 앱·DB, 백업 여유 |
| 제외 | RDS, 로드밸런서, NAT Gateway, 공개 관리자 화면 |

관리 화면은 인터넷에 열지 않고 SSH 터널로 본다. DB dump/restore, 자동 재기동,
SSM 비밀 주입, heartbeat, 로컬 8020 종료까지 끝나야 단일 운영 주체 전환이다.

## 7. 비용 전망

OpenAI Standard 단가로 보정한 현재 일일 전체 판단 기준선은 약 `$0.092`다.
장중 sweep과 중요 사건 재판단을 포함한 예상은 다음과 같다.

| 상황 | 하루 | 10거래일 |
|---|---:|---:|
| 조용함 | `$0.25~0.30` | `$2.5~3.0` |
| 보통 | `$0.40~0.50` | `$4.0~5.0` |
| 매우 바쁨 | `$0.70~0.90` | `$7.0~9.0` |
| 코드상 하드캡 | `$3` | `$30` |

예상 중심값은 하루 `$0.45`, 10거래일 `$4.5`다. 기사 전부를 LLM으로 요약하지
않고 결정론 필터를 통과한 관련 종목만 판단하는 전제다.

AWS 최소안까지 합치면 보통 10일 총 운영비는 약 `$8.5~10`이다. 환율
`1 USD=1,400원`의 예산용 가정으로 약 1.2만~1.4만원이다.

## 8. 최종 완료 정의

다음이 모두 충족되어야 MVP-2의 이번 범위를 끝났다고 본다.

- 관리자·사용자 화면의 숫자가 DB 원장과 일치한다.
- 관리자가 계좌를 개설·성향 변경·정지할 수 있고 사용자는 쓰기 권한이 없다.
- 일일 전체 체인은 같은 슬롯에 한 번만 돈다.
- 정규장 가격 방어가 1분 안에 손절·익절을 평가한다.
- 새 중요 공시·뉴스가 목표 주기 안에 들어와 관련 종목만 재판단한다.
- 무관·중복 사건은 LLM 호출과 주문을 만들지 않는다.
- cooldown·예산 예약·하드캡·주문 멱등성이 함께 지켜진다.
- 앱/DB 중단은 Healthchecks.io가, 내부 잡 실패는 앱 Telegram이 구분해 알린다.
- 실제 정규장 2회에서 중복 LLM·중복 주문 없이 완주한다.
- 현재 변경을 비밀 검사한 뒤 커밋하고, push는 사용자가 지시할 때만 한다.
