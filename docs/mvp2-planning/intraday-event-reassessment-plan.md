> **상태: 설계·견적 확정안 / 구현 전**
>
> 지금 구현된 장중 감시는 **가격을 1분마다 읽는 뼈대**까지다. 일일 JOB 체인은
> 거래일마다 한 번만 돌고, `disclosures`·`news`·`news_wire`도 그 체인에 묶여

## 0. 결론 먼저

전체 JOB 체인을 여러 번 돌리는 방식은 맞지 않는다. 상장 목록·일봉·전 종목
스크리닝·배분까지 매번 반복하면 같은 데이터를 다시 읽고 같은 판단을 다시
사게 된다. **무거운 일일 체인은 1회 유지하고, 장중에는 신규 사건만 증분 수집해
영향받은 종목만 재판단**하는 것이 최종 구조다.

| 영역 | 현재 | 최종 권고 | LLM 호출 |
|---|---|---|---|
| 전체 JOB 체인 | 거래일 1회 | **거래일 1회 유지** | 일일 분석 1회 |
| 상장 목록 `universe` | 7일마다 | 7일마다 유지 | 없음 |
| 일봉·거시·스크리닝 | 거래일 1회 | 거래일 1회 유지 | 스크리닝 자체는 없음 |
| 공시 `disclosures` | 거래일 1회 | 프리마켓·정규장 **30분마다 증분 수집** | 새 중요 공시가 있을 때만 |
| 뉴스 `news` | 거래일 1회 | 프리마켓·정규장 **15분마다 증분 수집** | 새 중요 뉴스가 있을 때만 |
| 와이어 `news_wire` | 거래일 1회 | 프리마켓·정규장 **10분마다 증분 수집** | 새 중요 뉴스가 있을 때만 |
| 가격 감시 | 정규장 1분 폴링 | 그대로 유지 | 가격 트리거 때만 |
| 정기 재판단 | 설정은 있으나 꺼짐 | NY 10:00 / 12:45 / 15:15 | 대상 종목만 |
| 주문·청산 | 일일 체인 중심 | 판단이 바뀐 종목만 표적 실행 | 없음 |

핵심 규칙은 **새 사건 0건이면 장중 사건 때문에 발생하는 LLM 호출도 0건**이다.

### `news`와 `news_wire`의 차이

- `news`는 Alpaca가 제공하는 Benzinga 계열 구조화 뉴스다. 기사 ID와 연결
  종목이 비교적 명확해서 주 뉴스 소스로 쓴다.
- `news_wire`는 기업·기관이 RSS/Atom으로 직접 내보내는 보도자료 피드를
  보완 수집하는 경로다. Alpaca 뉴스에 늦게 들어오거나 빠진 발표를 잡는
  **보조망**이지 별도의 전략가가 아니다.
- 같은 소식이 두 경로에 들어올 수 있으므로 원문 URL·제목·발행 시각으로
  중복 제거한 뒤 한 사건으로 취급한다.

와이어를 빼도 핵심 시스템은 동작하지만 정보 누락 가능성이 커진다. 이미 수집기가
있으므로 없애기보다 **10분 간격의 저비용 보조망으로 유지**하는 편이 낫다.

## 1. 왜 JOB 체인 전체를 반복하지 않는가

현재 러너는 60초마다 깨어나지만 `tb_job_run`과 잡별 cadence가 같은 뉴욕 슬롯의
중복 실행을 막는다. 선언이 없는 잡은 기본 거래일 1회다. 이 구조 덕분에 앱이
계속 살아 있어도 일일 체인이 폭주하지 않는다.

문제는 중복 방지 자체가 아니라 **변하는 데이터와 변하지 않는 데이터를 같은
체인으로 묶은 것**이다.

- 상장 목록·일봉·거시·전체 스크리닝은 장중 매분 다시 할 이유가 없다.
- 공시·뉴스는 장중에도 계속 생기므로 하루 1회로는 늦다.
- 전략가는 모든 종목을 다시 볼 필요 없이, 새 사건과 직접 관련된 보유 종목·
  당일 후보만 다시 보면 된다.
- allocation 전체를 다시 돌리지 않고, 판단이 실제로 바뀐 종목만 기존 멱등
  주문 경로로 보낸다.

따라서 최종 런타임은 둘로 나뉜다.

```text
일일 1회
  수집 기준선 → 전체 스크리닝 → 전략가/비평가 → 청산 → 배분

장중 반복
  공시·뉴스 증분 수집 → 중복 제거 → 중요도/종목 연결
      ├─ 새 중요 사건 없음 → 종료 (LLM 0콜)
      └─ 있음 → 해당 보유/당일 후보만 전략가+비평가
                    → 판단 변경분만 표적 청산/주문

장중 가격 방어
  1분 시세 → 보호성 청산 → ±5% 또는 정기 sweep 재판단
```

## 2. 구현 범위와 완료 기준

### 2-1. 증분 수집

각 소스는 마지막 성공 cursor/watermark 이후만 읽는다. SEC accession number,
Alpaca/Benzinga news ID, wire 원문 URL·발행 시각 등 공급자 고유 키를 우선하고,
없을 때만 안정된 복합 키를 쓴다.

- 같은 문서를 여러 폴링에서 받아도 DB에는 한 번만 저장한다.
- 네트워크 실패 시 cursor를 전진시키지 않는다.
- 재기동 후 마지막 성공 지점부터 이어 받는다.
- 늦게 도착한 기사 때문에 짧은 overlap을 다시 읽더라도 unique key가 중복을
  막는다.

### 2-2. 사건 라우팅

수집 직후 싼 결정론 필터를 먼저 적용한다.

1. 새 문서인가
2. 신뢰할 수 있는 출처인가
3. 티커를 확정적으로 연결할 수 있는가
4. 보유 종목 또는 당일 후보인가
5. 실적·가이던스·M&A·규제·증자·중대 내부자 거래처럼 재판단할 사건인가

여기서 탈락하면 LLM을 부르지 않는다. 애매한 소수만 경량 분류 모델을 선택적으로
쓸 수 있지만, MVP 기본값은 결정론 필터다. 전략가·비평가는 통과한 종목만 기존
공용 분석 경로로 부른다.

### 2-3. 안전장치

- 같은 `event_id + ticker + persona`는 한 번만 판단한다.
- 종목별 30분 cooldown을 가격·뉴스·공시 트리거가 함께 사용한다.
- 매도성 판단을 위해 일일 LLM 예산의 20%를 예약한다.
- `budget.daily_llm_usd`의 하드캡은 계속 `$3`이며, 초과 시 **새 매수 판단부터
  멈추고 보호성 청산은 막지 않는다.**
- 텔레그램은 매 폴링 성공을 보내지 않는다. 중요 사건 발견·재판단·실패·장시간
  무수신만 알린다.

### 2-4. 관제실에서 보여야 하는 것

- 소스별 최근 폴링·성공·cursor·신규/중복/실패 건수
- 사건별 연결 티커·필터 결과·재판단 여부
- 재판단 이유(`news`, `disclosure`, `price_move`, `scheduled_sweep`)
- 종목별 cooldown과 오늘 사용/예약 LLM 예산
- “폴링은 정상이나 새 사건 0건”과 “폴링 자체가 멈춤”의 구분

## 3. 개발 기간 — 2일 압축안

핵심 기능은 **2개 개발일 안에 구현 가능**하다. 앞서 적은 6~7.5일은 관제실
세부 화면, 장애 복구 확장, 문서 정리와 실제 장중 관측까지 각각 독립 작업으로
잡은 보수적 견적이었다. 현재 코드에는 수집기·전략가/비평가·장중 러너·예산
방어선이 이미 있으므로, 새 추상화를 넓히지 않고 그 사이만 연결하면 된다.

| 날 | 구현 범위 | 완료 조건 |
|---|---|---|
| **1일차** | 요율 정정, source별 cursor·중복 방지, 공시 30분·뉴스 15분·와이어 10분 증분 실행 | 재기동·중복 수신에도 신규 사건이 한 번만 저장됨 |
| **2일차** | 보유+당일 후보 라우팅, 기존 전략가/비평가 연결, 30분 cooldown·예산·표적 실행, 핵심 로그·텔레그램 | mock 압축 주기에서 무사건 0콜·관련 사건 1회 재판단·중복 주문 없음 |

2일 안에 포함하지 않는 것은 관제실의 상세 사건 탐색 UI, 스트리밍 전환,
AWS 이전이다. 관제실에는 우선 최근 폴링·신규·중복·실패·재판단 결과만 붙인다.
실제 NYSE 정규장 확인은 코드 완성과 별개로 **다음 거래일 1회 관측**이 필요하고,
두 번째 거래일은 안정성 확인이다. 즉 코드는 2일에 끝내되 운영 확정은 실장
관측 뒤에 한다.

## 4. LLM 비용

### 4-1. 확인된 기준선

2026-07-22 `tb_llm_usage` 실측은 다음과 같다.

| 항목 | 값 |
|---|---:|
| 전략가 | 22콜 · 입력 45,270 · 출력 5,054 토큰 |
| 비평가 | 21콜 · 입력 25,221 · 출력 3,691 토큰 |
| 원장 기록액 | `$0.046110` |
| 공식 Standard 요율로 보정한 실제 추정 | **`$0.092220`** |

여기에는 바로잡아야 할 설정 차이가 있다. 현재 `pipeline.yaml`의
`gpt-5.4-mini` 단가는 입력 `$0.375` / 출력 `$2.25` per 1M으로, OpenAI 공식
표의 **Batch/Flex** 요율이다. 앱은 `service_tier`를 지정하지 않아 Standard로
호출하며, 2026-07-23 공식 Standard는 입력 `$0.75` / 출력 `$4.50`이다.
따라서 구현 첫 작업에 설정 단가를 Standard로 고치고 예산 원장 회귀 테스트를
넣는다. 공식 가격은 바뀔 수 있으므로 배포 시
[OpenAI API Pricing](https://developers.openai.com/api/docs/pricing)과 다시 맞춘다.

실측 한 바퀴를 22개 대상에 나누면 전략가+비평가 한 종목 재판단은 평균
약 **`$0.0042`**다. 아래 견적은 이 실측 단가를 쓴다.

### 4-2. 완성 후 하루·10거래일 예상

가정:

- 일일 전체 분석 1회: `$0.092`
- 정기 sweep 3회
- 중요 사건은 결정론 필터 뒤 관련 종목만 LLM에 전달
- 재시도·출력 편차 여유 25% 포함

| 운용일 | 대상 규모 예시 | 하루 | 10거래일 |
|---|---|---:|---:|
| 조용한 날 | sweep당 10종목, 사건 0~3종목 | **`$0.25~0.30`** | **`$2.5~3.0`** |
| 보통 날 | sweep당 20종목, 사건 3~8종목 | **`$0.40~0.50`** | **`$4.0~5.0`** |
| 매우 바쁜 날 | sweep당 30종목, 사건 20~50종목 | **`$0.70~0.90`** | **`$7.0~9.0`** |
| 코드상 절대 상한 | `daily_llm_usd=3.0` | **최대 `$3`** | **최대 `$30`** |

따라서 현재 예상의 중심값은 **하루 약 `$0.45`, 10거래일 약 `$4.5`**다.
실제 비용은 기사 수보다 “필터를 통과한 종목 수”에 비례한다. 모든 기사마다
LLM 요약을 붙이면 이 견적은 무효가 되므로 그렇게 구현하지 않는다.

## 5. AWS 최소 비용 구성

### 5-1. 권고 최소안 — Lightsail 한 대

10일 내외 MVP 관측에는 RDS, 로드밸런서, NAT Gateway, Kubernetes가 필요 없다.

```text
Lightsail Small 2GB (public IPv4 포함)
  ├─ quantinue 앱 (systemd 또는 Docker)
  ├─ PostgreSQL 컨테이너
  ├─ 로컬 볼륨 + 일일 압축 백업
  └─ 외부 API / Telegram / healthchecks.io

관리 화면: 8020 인터넷 공개 금지 → SSH 터널
비밀: SSM Parameter Store Standard 또는 root 전용 환경 파일
```

[AWS Lightsail 공식 표](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-bundles.html)의
public IPv4 포함 Small은 2 vCPU, 2GB RAM, 60GB SSD, 3TB 전송량에 **월 `$12`**다.
1GB `$7`도 숫자상 싸지만 앱·PostgreSQL·배포 순간 메모리 여유가 작아 운영
최소로 권하지 않는다. IPv6-only `$10`은 일부 외부 공급자의 IPv6 지원 여부를
별도로 검증해야 하므로 `$2`를 아끼기 위해 선택하지 않는다.

| 항목 | 월 | 10일 환산 | 비고 |
|---|---:|---:|---|
| Lightsail Small 2GB | `$12` | 약 `$4.0` | IPv4·60GB SSD 포함 |
| 백업 여유 | `$1~3` | `$0.3~1.0` | 보관량·방식에 따라 변동 |
| healthchecks.io | `$0` | `$0` | 현재 무료 체크 사용 전제 |
| 도메인·TLS·로드밸런서 | `$0` | `$0` | 관리 화면을 SSH 터널로만 봄 |
| **인프라 합계** | **`$13~15/월`** | **`$4.3~5.0/10일`** | 세금·환율 제외 |

환산 편의를 위해 `1 USD = 1,400원`을 가정하면 월 **약 1.8만~2.1만원**,
10일 **약 6천~7천원**이다. 환율 실시간값이 아니라 예산용 가정이다.

LLM 중심값까지 합치면 10거래일 총 운영비는 대략
**`$8.5~10`(약 1.2만~1.4만원)**이다. 바쁜 날이 계속되면 LLM을 포함해
약 `$14` 안팎, 하드캡을 매일 모두 쓰는 비정상 최악은 약 `$35`다.

### 5-2. EC2를 꼭 쓸 때

EC2 `t4g.small` + gp3 + public IPv4도 가능하지만 인스턴스·디스크·IPv4를
따로 계산해야 한다. AWS는 public IPv4에 시간당 `$0.005`를 부과하므로 이것만
월 약 `$3.60`이다([Amazon VPC Pricing](https://aws.amazon.com/vpc/pricing/)).
이 규모에서는 Lightsail의 묶음 가격과 운용 단순성이 낫다. SSM Parameter
Store Standard는 추가 요금이 없다
([AWS Systems Manager Pricing](https://aws.amazon.com/systems-manager/pricing/)).

RDS를 추가하면 장애 격리는 좋아지지만 MVP 비용 최소 목표와 맞지 않는다.
장기 운영을 결정하고 복구 시간 목표가 생긴 뒤 분리한다.

### 5-3. AWS 구축 시간

| 작업 | 예상 |
|---|---:|
| 인스턴스·방화벽·IAM·비밀 주입 | 0.5일 |
| 앱/DB 배포·dump/restore·자동 재기동 | 0.5~1.0일 |
| 백업·heartbeat·로그·SSH 터널 | 0.5일 |
| 실제 슬롯 전환 검증과 로컬 중복 실행 방지 | 0.5일 |
| **합계** | **1.5~2.5 개발일** |

AWS 서비스 자체의 초기 구축비는 `$0`이고, 위 시간은 개발·검증 인건비다.

## 6. 권고 구현 순서

1. **요율 정정** — `gpt-5.4-mini` Standard 단가와 예산 원장 테스트.
2. **증분 원장** — source별 cursor, event unique key, 재기동 복구.
3. **수집 분리** — disclosures 30분, news 15분, wire 10분.
4. **사건 라우터** — 결정론 필터, 티커 연결, 보유+당일 후보 교집합.
5. **표적 재판단** — 기존 전략가·비평가 경로 재사용, 30분 cooldown.
6. **표적 실행** — 판단 변경분만 청산/주문, 기존 멱등성 유지.
7. **관측** — 관제실, 텔레그램, heartbeat, 실패와 무사건 구분.
8. **mock 검증** — 압축 주기로 중복·재기동·네트워크 실패를 먼저 확인.
9. **실장 검증** — 실제 NYSE 정규장 2회, 그 뒤 `rejudge.enabled=true`.
10. **AWS 이전** — 로컬 실장 검증 후 단일 실행 주체로 전환.

## 7. 완료 판정

다음이 모두 실측되어야 “하루 1회 문제를 해결했다”고 말한다.

- 일일 전체 체인은 같은 슬롯에 여전히 한 번만 돈다.
- 정규장 중 새 공시·뉴스가 다음 폴링 주기 안에 DB에 들어온다.
- 같은 사건을 반복 수신해도 한 번만 재판단한다.
- 무관한 기사와 기존 기사는 LLM 비용을 만들지 않는다.
- 관련 사건은 해당 종목의 전략가·비평가만 깨운다.
- cooldown, 20% 매도 예약, `$3` 하드캡이 함께 지켜진다.
- 재기동·일시 네트워크 단절 뒤 cursor 유실 없이 이어진다.
- 관제실과 텔레그램에서 “정상 무사건”과 “수집 중단”을 구분할 수 있다.
- 실제 정규장 2회 동안 중복 주문·중복 LLM 호출이 없다.
